# TOPIK AI Admin 페이지별 테이블/필드 정의

이 문서는 관리자 페이지의 `테이블 컬럼`, `입력 필드`, `필터`, `정렬`, `액션`을 정리합니다.  
페이지 목적과 상위 상태/유형 분류는 `docs/architecture/admin-overview.md`, 화면 블록과 미확정 정책은 `docs/specs/page-ia/*.md`를 먼저 참고합니다.

## 공통 규칙

- 페이지 단위 탭, 검색, 필터, 요약 액션, 주요 버튼은 반드시 본문 컨테이너 안에 배치합니다.
- 탭, 필터, 주요 버튼을 카드 헤더 `extra`나 본문 밖 별도 줄에 분리하지 않습니다.
- 연결 가능한 식별자(ID, 작성자, 결제 번호)는 링크 스타일 텍스트를 우선 사용합니다.
- 사용자 관련 클릭 이벤트에서 `사용자명`은 항상 공통 `table-navigation-link` 스타일(파란색 링크 텍스트)로 표시하고, 클릭 시 `회원 상세`로 이동합니다.
- 링크 대상이 없는 상세 정보는 Drawer 또는 모달로 확인합니다.
- placeholder 페이지는 실제 컬럼을 확정하지 않고 `예정 데이터 블록` 수준으로만 기록합니다.
- 파괴적 액션은 확인 단계와 사유 입력 수단을 함께 설계합니다.
- 테이블 `더보기` 메뉴에서는 일반 액션을 상단 목록에 두고, `삭제`와 `정지 해제`가 아닌 `정지` 계열 액션은 반드시 드롭다운 하단 분리 영역의 독립 버튼으로 노출합니다.
- 목록 운영형 페이지는 `Card` 본문 내부에 `본문 상단 UI -> 안내 문구(선택) -> Table` 순서를 유지합니다.
- 목록 운영형 페이지의 본문 컨테이너는 공용 `AdminListCard`를 기본값으로 사용하고, `toolbar`에는 탭/검색/총 건수/핵심 액션을 함께 배치합니다.
- 목록 운영형 페이지에서 `AdminListCard` 안에 가로 스크롤 테이블(`AdminDataTable`)을 배치할 때는, 테이블 viewport와 하단 가로 스크롤 영역이 카드 하단까지 이어지도록 유지합니다.
- 가로 스크롤이 필요한 경우에도 테이블 wrapper와 pagination은 카드 본문 너비 밖으로 밀려나지 않아야 하며, 초과 폭은 테이블 내부 스크롤에서만 처리합니다.
- `SearchBar`는 `검색 대상 선택 Select + 검색 Input + 상세 버튼 + 총 건수` 구조를 기본값으로 사용합니다.
- `등록/추가/생성/새로 만들기` 계열 버튼은 본문 기준 테이블 또는 주요 데이터 블록의 우측 상단에 배치합니다.
- 본문 상단의 `등록/추가/생성/새로 만들기` 계열 버튼은 항상 Ant Design `large` 크기를 사용합니다.
- `SearchBar`를 쓰는 페이지는 `SearchBar`와 `등록/추가/생성` 버튼을 같은 줄에 두고, 버튼을 우측 끝에 배치합니다.
- 채널/모드 탭이 필요한 목록형 페이지도 카드 header `extra`나 본문 밖 줄이 아니라 `AdminListCard.toolbar` 안에서 `SearchBar` 위쪽 행으로 정렬합니다.
- `SearchBar`가 없는 목록형 페이지는 본문 상단 오른쪽 정렬 툴바에서 `총 n건` 요약과 `등록/추가/생성` 버튼을 같은 줄에 두고, 건수는 버튼 왼쪽에 배치합니다.
- 상세 필터와 `필터 초기화`/`적용` 버튼은 `SearchBar`의 `상세 검색` 드롭다운 안에서 같은 위치 규칙으로 유지합니다.
- `상세 검색`에는 페이지 기준 날짜 필드의 기간만 두고, `정렬`, `상태`, `카테고리`, `역할`, `수단` 같은 비기간 항목은 넣지 않습니다.
- `상세 검색` 드롭다운에는 페이지 기준 날짜 필드를 대상으로 한 `DatePicker.RangePicker` 기간 필터를 항상 포함합니다.
- 드롭다운 안의 기간 값 변경은 draft로만 유지하고, 테이블 재조회와 URL 갱신은 `적용` 버튼 클릭 시점에만 수행합니다.
- `page`, `pageSize`를 URL/store와 함께 관리하는 목록은 페이지네이션 변경 시 목록을 다시 조회하고, 테이블 위에 loading 애니메이션을 표시합니다.
- 페이지네이션 재조회 중에는 직전 성공 데이터를 유지하고, 실패해도 화면 전체를 비우지 않습니다.
- 단순한 관련 화면 이동은 우측 상단 버튼보다 안내 문구 링크나 테이블 셀 링크를 우선 사용합니다.
- 컬럼 필터는 코드값, 정책값, 상태값처럼 운영 기준이 이미 정해진 고정값 집합 컬럼에만 둡니다.
- 제목, 이름, ID, 자유 텍스트, 숫자, 금액, 건수, 비율, 날짜/시각 컬럼은 컬럼 필터를 두지 않고 정렬과 상단 검색/기간 필터로만 제어합니다.
- 상태 컬럼 필터도 고정된 상태 세트가 있고 운영상 좁혀볼 목적이 분명할 때만 유지하며, URL/store 동기화가 걸린 제어 필터는 반드시 유지합니다.
- 상태값이 정확히 2개인 테이블 컬럼이라도 관리자가 해당 화면에서 직접 조치할 수 있는 상태일 때만 스위치 컴포넌트를 사용합니다. 조회 전용 상태, 외부 원천 데이터, 간접 관리 상태는 배지 또는 텍스트로 유지합니다.
- 상태 전환을 스위치로 옮긴 화면에서는 동일한 `숨김/비활성/비공개/처리 완료` 계열 항목을 액션 컬럼 `더보기` 메뉴에 중복 노출하지 않습니다.
- `더보기` 정리 후 실질 액션이 1개만 남는 액션 컬럼은 드롭다운 대신 직접 실행 가능한 텍스트 또는 아이콘 버튼을 사용합니다.
- 상태나 레벨처럼 정해진 값 집합을 보여주는 컬럼 제목에는 안내 아이콘 툴팁을 두고, 가능한 상태값과 각 의미를 함께 노출합니다.
- 입력 Modal, Drawer, 보조 패널의 항목형 UI는 `Descriptions` 기반 입력 테이블을 기본값으로 사용하고, `Form`은 검증과 상태 관리 용도로만 감쌉니다. 단일 본문 에디터처럼 표 구조가 맞지 않는 캔버스형 입력만 예외로 둡니다.
- `Descriptions` 기반 입력 테이블에서 필수 검증(`required`)이 걸린 항목은 label(`th`) heading에 빨간 `*`를 표시해 향후 DB/API 필수 필드 후보임을 운영자가 즉시 인지할 수 있게 합니다.
- 행 클릭 상세 Drawer의 헤더/본문/푸터 슬롯 규칙은 `docs/guidelines/admin-detail-drawer-guidelines.md`를 단일 기준으로 사용합니다.
- 행 클릭 상세 Drawer 헤더 오른쪽은 상태/태그 같은 읽기 전용 메타만 두고, 전역 조치 버튼은 푸터 오른쪽에 둡니다.
- 행 클릭 상세 Drawer 푸터 왼쪽은 `감사 로그 확인`, 오른쪽은 주요 조치 버튼 묶음을 기본 위치로 사용합니다.
- 행 클릭 상세 Drawer의 header/footer 시각 baseline은 `Community > 게시글 관리 > 게시글 상세 Drawer`를 따르고, header 메타 간격 `8px`, footer 버튼 간격 `8px`, footer wrap 구조, footer 액션 버튼 `large` 크기를 기본값으로 사용합니다.
- 작성/편집/조건 설정 Drawer도 header/footer chrome은 같은 baseline을 따르고, footer는 왼쪽 `취소`, 오른쪽 보조/주요 액션 버튼 묶음, 버튼 `large`, gap `8px`, wrap 허용 구조를 기본값으로 사용합니다.
- `Drawer` 내부 최상위 테이블은 첫 번째 열을 좌측 고정하고, 본문 높이는 최대 5개 행 정도를 기준으로 유지하며, 페이지네이션은 오른쪽 아래에 둡니다.
- `Drawer` 내부 expandable row에 렌더되는 하위 테이블은 첫 번째 열을 고정하지 않고, 페이지네이션 없이 한 번에 표시합니다.
- `Drawer` 내부 섹션 간 간격은 `Space` 기준 `32px`를 기본값으로 사용합니다.
- `DetailDrawer` 내부 `Descriptions`의 label(`th`) 폭은 전역 `130px`로 통일하고, 개별 페이지의 inline width override는 예외 사유가 있을 때만 허용합니다.
- 위 Drawer 레이아웃 규칙이 바뀌면 `docs/guidelines/admin-detail-drawer-guidelines.md`와 영향받는 `docs/specs/page-ia/*.md`를 함께 갱신합니다.

## 1) 회원 관리 > 회원 목록

- 테이블 컬럼: 회원 ID, 이메일, 닉네임, 가입일, 최근 접속, 회원 상태, 등급, 구독 상태
- 필터: 검색어, 가입 기간
- 정렬: 가입일, 최근 접속, 회원 상태
- 액션: 회원 상세, 회원 정지/해제, 관리자 메모
- 레이아웃 메모: 요약 없는 목록 운영형 기준으로 `Card 내부 SearchBar -> Table` 순서를 유지합니다.
- 상태 표현 메모: `구독 상태` 컬럼은 외부 구독 원천 데이터를 보여주는 정보용 컬럼이므로 스위치 조치 없이 텍스트 상태로 표시합니다.

## 2) 회원 관리 > 강사 관리

- 현재 상태: 구현됨
- 구현 컬럼: 강사 ID, 이름, 이메일, 소속, 담당 국가, 계정 상태, 활동 상태, 담당 과정 수, 담당 학습자 수, 최근 활동, 최근 조치일
- 구현 필터: 검색어(강사명/강사 ID/이메일/대상 그룹), 최근 활동일
- 구현 액션: `대상 그룹 보기`, `강사 정지`, `강사 정지 해제`
- 상세 Drawer: 기본 정보, 전문 분야, 담당 과정, 최근 메시지 발송 이력, 관리자 메모, 감사 로그 확인(`Instructor`)
- Drawer 상세 테이블:
  - `담당 과정`은 `과정 ID`, `과정명`, `난이도`, `학습자 수`, `과정 상태` 테이블로 표시합니다.
  - `최근 메시지 발송 이력`은 `발송 ID`, `채널`, `제목`, `발송 시각`, `상태` 테이블로 표시합니다.
  - `관리자 메모`는 `메모 ID`, `작성 관리자`, `작성일`, `메모 요약` 테이블로 표시하고, 전체 메모 내용은 expandable row에서 확인합니다.
  - 위 3개 Drawer 테이블은 첫 번째 열 고정, 최대 5행 높이, 오른쪽 아래 페이지네이션 규칙을 공통 적용합니다.
- 예정 데이터 블록:
  - 강사 기본 정보: 강사 ID, 이름, 소속, 상태
  - 운영 정보: 담당 과정 수, 담당 학생 수, 최근 활동, 계약/정산 상태
  - 액션: 상태 변경, 담당 과정 연결, 메시지 발송

## 3) 회원 관리 > 추천인 관리

- 현재 상태: 구현됨
- 상세 IA 문서: `docs/specs/page-ia/users-referrals-page-ia.md`
- 구현 컬럼: 추천 코드, 추천인 회원, 피추천인 수, 추천 확정 수, 누적 보상, 최근 사용일, 코드 상태, 이상치 여부
- 구현 필터: 검색어(추천 코드/추천인 ID/추천인 이름), 최근 사용일, `코드 상태` 컬럼 필터(URL/store 동기화), `이상치 여부` 컬럼 필터(URL/store 동기화)
- 구현 액션: `포인트 관리 이동`, `보상 수동 조정`, `이상치 검토 완료`
- 상세 Drawer: 기본 정보, 추천 관계 및 보상, 정책 스냅샷, 이상치 및 운영 메모, 감사 로그 확인(`Referral`)
- 정책 메모:
    - 추천 확정 시점, 보상 수단, 수동 보정 권한, 회수/취소 규칙이 아직 미확정이므로 `누적 보상`과 보상 원장 포맷은 가정값 기반입니다.
    - 추천인 회원 1명당 추천 코드 1개를 운영 규칙으로 고정합니다.
    - `추천인 회원` 셀은 사용자 이름만 노출하고, ID/닉네임/순번 같은 보조 식별자는 함께 표시하지 않습니다.
    - 상세 Drawer 진입은 `더보기 > 상세 보기`가 아니라 행 클릭으로 처리합니다.
    - Drawer 안 `추천 관계 및 보상`은 피추천인 1명당 1개 행으로 구성한 관계 테이블을 사용하고, 사용자명은 공통 `table-navigation-link` 스타일을 사용합니다.
    - 관계 테이블 컬럼은 `이름 링크`, `진행 시각(가입/확정)`, `관계 상태`, `검수`, `누적 보상`을 기본으로 둡니다.
    - 각 관계의 보상 원장은 행 확장 영역의 하위 테이블로 표시하고, 하위 테이블 컬럼은 `유형`, `금액`, `처리 시각`, `사유` 중심으로 최소화합니다.
    - 관계 테이블은 첫 번째 열 고정, 최대 5행 높이, 오른쪽 아래 페이지네이션 규칙을 적용합니다.
    - 하위 원장 테이블은 첫 번째 열을 고정하지 않고, 페이지네이션 없이 한 번에 표시합니다.
    - 관계에 귀속되지 않는 보상 조정이 있을 때만 `코드 단위 조정` 테이블을 별도로 노출합니다.
    - `정상`은 관계 상태값이 아니라 기본 검수 메모였으므로, 현재 UI에서는 의미 있는 메모가 있을 때만 노출합니다.
    - 감사 로그 딥링크는 현재 `Referral + 추천 코드 ID` 기준으로 동작하며, 조정/검수 개별 ID 분리는 후속 정책 확정 시 재검토가 필요합니다.
- 상태 표현 메모: `코드 상태` 컬럼은 `활성/비활성` 스위치로 조치하고, 기존 비활성화/재활성화 더보기 메뉴는 제거합니다.

## 4) 커뮤니티 > 게시글 관리

- 구조 메모: 상세 Drawer `설정 구조`는 Ant Design `Tree`를 그대로 사용하고, 루트 아래에 운영 값 노드와 각 계층 마지막 `추가` 노드를 함께 둡니다. 운영 값 노드는 클릭 시 수정, 드래그 시 순서 변경, `추가` 노드는 같은 레벨의 새 운영 값 추가 진입점입니다.
- 정렬 메모: `지금 운영 중인 값` 테이블은 왼쪽 드래그 핸들로 행 순서를 바꾸며, 순서 변경은 즉시 저장되고 `item_reordered` 이력과 감사 로그를 남깁니다.
- 검증 메모: 운영 값 등록/수정 Modal은 현재 mock 데이터 기준으로 같은 설정 그룹 안의 `운영 값 코드`, `운영 값 라벨` 중복을 즉시 검사하고, 저장 시 service에서도 같은 조건을 다시 검증합니다.
- 사용자 영향 메모: 사용자 영향 상태는 화면에서 `사용자 노출 확인`, `사용자 노출 추정`, `내부 전용`, `노출 예정`으로 풀어 쓰고, Tooltip에 각 상태 의미를 함께 설명합니다.
- URL 복원 메모: `summaryFilter`, `searchField`, `keyword`, `startDate`, `endDate`, `selected` 쿼리로 목록/상세 상태를 복원합니다.

## 39) System > 메타데이터 관리 > Tree 삭제/삭제 Modal 보강
- `설정 구조`는 `Tree` 기본 라인과 확장/축소 아이콘을 유지하고, 운영 값 노드 hover 시 텍스트 우측에만 삭제 아이콘을 노출합니다.
- Tree 각 계층의 마지막 `추가` 노드는 같은 깊이의 운영 값 추가 진입점으로 유지합니다.
- `운영 값 수정` Modal 하단에는 `운영 값 삭제` 버튼을 두고, 클릭 시 기존 수정 Modal을 닫은 뒤 `ConfirmAction`을 열어 z-index 충돌 없이 삭제를 진행합니다.
- 운영 값 삭제 결과는 `지금 운영 중인 값` 테이블, `설정 구조` Tree, 변경 이력, 감사 로그에 동시에 반영합니다.
