# TOPIK AI Admin 페이지별 테이블 설계

본 문서는 `docs/architecture/admin-information-architecture.md` 기준 메뉴를 그대로 따릅니다.

## 1) Users > 회원 목록

### 테이블 컬럼
| 컬럼 | 설명 |
| --- | --- |
| User ID | 회원 ID |
| 이메일 | 로그인 이메일 |
| 닉네임 | 사용자 이름 |
| 가입일 | 회원 가입 날짜 |
| 최근 접속 | 마지막 로그인 |
| 회원 상태 | 정상 / 정지 / 탈퇴 |
| 등급 | 사용자 등급 |
| 구독 상태 | 구독 여부 |

### 필터
- 이메일 검색
- 닉네임 검색
- 회원 상태
- 가입일 기간
- 구독 상태

### 액션
- 회원 상세 보기
- 회원 정지
- 회원 정지 해제
- 관리자 메모 작성

## 10) System > 권한 관리

### 테이블 컬럼
| 컬럼 | 설명 |
| --- | --- |
| 관리자 ID | 권한 대상 관리자 계정 |
| 이름 | 관리자 이름 |
| 상태 | 활성 / 비활성 |
| 역할(Role) | SUPER_ADMIN / OPS_ADMIN / CS_MANAGER / READ_ONLY |
| 권한 수 | 현재 보유 권한 개수 |
| 최근 수정 | 마지막 권한 변경 시각 |

### 액션
- 권한 부여
- 권한 수정
- 권한 회수(삭제)

### 권한 범위 설명(필수 노출)
- 권한 코드
- 권한명
- 모듈
- 권한 범위 설명(해당 권한으로 가능한 화면/액션 범위)
- 위험도(High / Medium / Low)

### 조치 기록 규칙
- 권한 변경(부여/수정/회수)은 사유 입력을 필수로 받는다.
- 변경 전/후 Role 및 권한 목록을 감사 로그에서 비교 가능해야 한다.
- `Target Type=Admin`, `Target ID=<관리자 ID>` 기준으로 감사 로그 역추적이 가능해야 한다.

## 11) System > 감사 로그

감사 로그 상세는 `docs/specs/admin-action-log.md`를 기준으로 관리합니다.

### 필수 연결 규칙
- Users/Community/Operation/Billing에서 발생한 데이터 변경 액션은 감사 로그에 기록
- 감사 로그에서 대상 ID를 통해 원본 화면으로 역추적 가능해야 함
