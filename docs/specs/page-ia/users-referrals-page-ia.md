# Users > 추천인 관리 상세 IA

## 1. 문서 목적

- 추천인 관리 페이지가 어떤 운영 데이터와 조치 흐름을 가져야 하는지 상세 수준에서 정리합니다.
- 아직 확정되지 않은 추천 정책과 보상 정책을 빈칸으로 두지 않고, 어떤 영향 때문에 추후 결정이 필요한지 문서로 남깁니다.
- 사용자 화면 개발이 시작될 때 추천 코드 입력, 보상 내역, 친구 초대/프로모션 화면이 관리자 데이터와 어떻게 연결될지 참고 자료로 사용합니다.

## 3. 페이지 목표와 비목표

### 목표

- 추천 코드, 추천 관계, 보상 상태를 한 화면에서 검색하고 검수합니다.
- 추천 보상과 관련된 운영 조치를 `상세 확인 -> 사유 입력 -> 감사 로그 확인` 흐름으로 관리합니다.
- 추후 사용자 화면에서 어떤 데이터와 상태를 노출해야 하는지 예측 가능한 기준 문서를 제공합니다.

### 비목표

- 사용자용 친구 초대 UI나 프로모션 UI를 직접 설계하지 않습니다.
- 추천 정책의 최종 비즈니스 결정을 이 문서에서 확정하지 않습니다.
- 회계/정산 시스템의 백엔드 규칙을 대체하지 않습니다.

## 4. 운영자 사용 시나리오

- 시나리오 1: 운영자가 추천 코드나 추천인 회원을 검색하고, 특정 코드의 상세 Drawer에서 추천 관계와 누적 보상을 확인합니다.
- 시나리오 2: 운영자가 이상치가 의심되는 추천 관계를 열어 추천 확정 상태와 보상 원장을 검수하고, 필요 시 수동 보정 또는 코드 비활성화 조치를 수행합니다.
- 시나리오 3: 운영자가 조치 후 `Target Type`, `Target ID` 기준으로 감사 로그를 확인하고, 필요 시 `커머스 > 포인트 관리` 또는 `회원 상세`로 이동해 후속 검증을 합니다.

## 6. 데이터 블록 정의

### 상단 요약 데이터

- `활성 코드 수`: 현재 사용 가능한 추천 코드 수. 사용자 가입/친구 초대 화면의 실제 사용 가능 코드 풀을 추정하는 기준입니다.
- `총 추천 수`: 추천 관계 총량. 추천 이벤트 운영 규모와 이상치 비율 분석의 기준입니다.
- `보상 지급 건수`: 지급이 확정된 보상 레코드 수. `커머스 > 포인트 관리`와의 정합성 검증 포인트입니다.
- `검수 필요 건수`: 이상치 플래그가 있거나 수동 보정 대기 상태인 건수. 운영 우선순위 큐 역할을 합니다.

### 검색/필터 데이터

- `추천 코드`: `keyword` 기반 검색. 사용자 초대 링크/입력 필드와 직접 연결될 후보 값입니다.
- `추천인 회원`: 추천인 사용자 ID 또는 이름 기준 검색. `Users > 회원 상세` 진입 기준이 됩니다.
- `상태`: 코드 상태 또는 추천 확정 상태를 의미합니다. 최종 상태 세트는 정책 확정 후 분리 정의가 필요합니다.
- `기간`: 추천 생성일 또는 최근 사용일 기준 범위 검색입니다.
- `이상치 여부`: 자기추천, 중복 기기, 비정상 반복 사용 등 검수 플래그를 보는 보조 필터입니다.

### 본문 데이터

- 기본 컬럼:
  - 추천 코드
  - 추천인 회원 이름
  - 피추천인 수
  - 추천 확정 수
  - 누적 보상
  - 최근 사용일
  - 코드 상태
  - 이상치 여부
- 운영 규칙:
  - 추천인 회원 1명은 추천 코드 1개만 가질 수 있습니다.
  - 따라서 목록의 1차 단위는 `추천 코드`이면서 동시에 `추천인 회원별 대표 추천 레코드`를 의미합니다.
- 테이블의 1차 단위는 문서/현재 스펙 기준으로 `추천 코드`를 권장합니다.
- 추천 관계 단위의 상세 정보는 Drawer 내부 관계 테이블과 확장 하위 원장 테이블로 구성합니다.

### 상세 Drawer 데이터

- `기본 정보`: 코드 ID, 추천인 회원, 생성일, 만료일, 현재 상태, 마지막 조치 시각
- `추천 관계 및 보상`: 피추천인 회원 목록, 가입/활성화 시각, 추천 확정 상태, 검수 메모, 관계별 누적 보상, 관계별 원장 이력
- `추천 관계 및 보상`은 피추천인 1명을 1개 행으로 보는 관계 테이블로 구성하고, 이름 클릭 시 `Users > 회원 상세`로 이동합니다.
- 피추천인 이름 링크는 테이블 사용자명과 동일한 공통 링크 스타일(`table-navigation-link`, 파란색 링크 텍스트)을 사용합니다.
- 관계 테이블의 기본 컬럼은 `이름 링크`, `진행 시각(가입/확정)`, `관계 상태`, `검수`, `누적 보상`을 포함합니다.
- 각 관계의 보상 원장은 행 확장 영역의 하위 테이블로 확인할 수 있어야 합니다.
- 하위 원장 테이블은 `유형`, `금액`, `처리 시각`, `사유`처럼 핵심 컬럼만 유지합니다.
- 관계 테이블, 하위 원장 테이블, `코드 단위 조정` 테이블은 첫 번째 열 고정, 최대 5행 높이, 오른쪽 아래 페이지네이션 규칙을 공통 적용합니다.
- 검수 메모는 이상치 또는 운영 검토 의견이 있을 때만 노출하고, 단순 `정상` 같은 기본 문구는 별도 표시하지 않습니다.
- 관계에 귀속되지 않는 보상 조정이 있을 때만 `코드 단위 조정` 테이블을 별도로 노출합니다.
- `정책 스냅샷`: 당시 적용된 보상 규칙, 조건, 수단, 버전
- `이상치/운영 메모`: 운영자가 남긴 검수 메모와 플래그 해제 이력

## 8. 상태값/정책/운영 규칙

| 항목 | 현재 상태 | 관리자 페이지 영향 | 사용자 화면 영향 | 추후 결정 필요 내용 |
| --- | --- | --- | --- | --- |
| 추천 확정 시점 | `미확정` | 추천 확정 수, 보상 지급 조건, 검수 필요 건수 계산식이 달라집니다. | 사용자에게 추천 성공/대기/실패를 어떤 시점에 보여줄지 달라집니다. | 가입 완료, 첫 결제, 첫 학습 완료 중 어떤 이벤트를 확정 기준으로 쓸지 결정 필요 |
| 보상 수단 | `미확정` | `커머스 > 포인트 관리` 연동 방식과 요약 카드 문구가 달라집니다. | 포인트 지갑, 쿠폰함, 현금성 리워드 내역 UI가 달라집니다. | 포인트/쿠폰/혼합 중 어떤 형태로 운영할지 결정 필요 |
| 수동 보정 권한 | `미확정` | 현재 `users.referrals.manage` 단일 권한으로 충분한지 판단이 필요합니다. | 사용자 보상 내역이 운영자 권한 정책에 따라 수정될 수 있습니다. | 보상 보정 전용 권한 분리 여부와 승인 체계 필요 |
| 회수/취소 규칙 | `미확정` | 환불/취소 시 보상 회수 로직과 상태 표시 기준이 필요합니다. | 이미 받은 혜택이 회수될 수 있는지, 어떤 문구로 안내할지가 달라집니다. | 환불/탈퇴/부정 사용 시 보상 회수 여부 결정 필요 |

- 현재 구현 상태값 세트:
  - 추천 코드 상태: `활성`, `비활성`
  - 이상치 검토 상태: `없음`, `검토 필요`, `검토 완료`
  - 추천 관계 상태: `대기`, `완료`, `취소`
  - 보상 원장 상태: `대기`, `완료`, `취소`
  - `정상`은 공식 상태값이 아니라 mock 검수 메모 기본값이었으며, 현재 UI에서는 별도 노출하지 않습니다.

## 10. 사용자 화면/B2C 영향 참고

| 사용자 화면 후보 | 영향 상태 | 이 페이지 데이터가 반영되는 방식 | 비고 |
| --- | --- | --- | --- |
| 가입/프로모션 화면의 추천 코드 입력 | `노출 예정` | 코드 유효/무효, 만료 여부, 오류 문구 기준이 이 페이지의 코드 상태와 연결됩니다. | 가장 직접적인 연결 |

| 보상 내역/리워드 히스토리 | `노출 예정` | 보상 원장과 확정 상태가 사용자 노출 내역의 원천이 됩니다. | 보상 수단 확정 후 구체화 필요 |

## 11. URL/상태 복원

- 기본 라우트: `/users/referrals`
- 필수 쿼리 파라미터 후보:
  - `page`
  - `pageSize`
  - `searchField`
  - `keyword`
  - `status`
  - `anomaly`
  - `startDate`
  - `endDate`
  - `selected`
- `status`, `anomaly`는 테이블 컬럼 필터와 URL/store가 동기화되어, 새로고침이나 공유 URL 진입 시 동일 조건으로 테이블이 자동 필터링되어야 합니다.
- `selected`는 행 클릭 Drawer 복원을 위해 유지하는 것을 권장합니다.
- 추천인 페이지는 상세 진입이 Drawer 구조이므로, 새로고침 후에도 동일한 검색 조건과 선택된 행이 복원되어야 합니다.

## 12. 네트워크 상태와 fail-safe

- `pending`: 상단 요약과 테이블은 loading 상태를 보여주고, 마지막 성공 데이터가 있으면 유지합니다.
- `success`: 정상 결과를 렌더링합니다.
- `empty`: 조건에 맞는 추천 데이터가 없음을 명확히 안내합니다.
- `error`: 조회 실패 안내, 오류 코드, 재시도 버튼, 마지막 성공 데이터 유지 문구를 제공합니다.
- 마지막 성공 상태 fallback: 에러 시 테이블 전체를 비우지 않고 직전 성공 데이터를 유지하는 방향을 권장합니다.
- 요청 취소/재시도: 페이지 이탈 시 abort, 조회 실패 시 retry 가능해야 합니다.

## 13. 구현 메모

- 현재 코드베이스에서 재사용할 컴포넌트:
  - `SearchBar`
  - `AdminListCard`
  - `AdminDataTable`
  - `ConfirmAction`
  - `AuditLogLink`
- 구현 파일:
  - `src/features/users/pages/users-referrals-page.tsx`
  - `src/features/users/model/referrals-types.ts`
  - `src/features/users/model/referrals-query-store.ts`
  - `src/features/users/api/referrals-service.ts`
  - `src/features/users/api/mock-referrals.ts`
- 감사 로그 연계 메모:
  - `Referral` targetType 라벨과 `/users/referrals?selected={codeId}` 딥링크가 구현되어, 추천인 페이지 조치 후 감사 로그에서 원본 화면으로 역이동할 수 있습니다.
- 구현 상태 메모:
  - 상단 요약 카드(`활성 코드 수`, `총 추천 수`, `보상 지급 건수`, `검토 필요 건수`)가 구현되었습니다.
  - 본문은 추천 코드 기준 테이블 + `status`/`anomaly` 컬럼 필터(URL/store 동기화) + 날짜 상세 검색 구조로 구현되었습니다.
  - 추천인 회원 컬럼은 사용자 이름만 노출하며, 이름 클릭 시 `회원 상세`로 이동합니다. 셀 내부에 ID, 닉네임, 순번 같은 보조 식별자는 노출하지 않습니다.
  - 행 클릭 시 상세 Drawer가 열리며, 액션 메뉴에서는 `상세 보기`와 중복된 추천인 이동 항목을 제거했습니다.
  - mock 데이터도 `추천인 1명당 코드 1개` 규칙에 맞춰 정리되었습니다.
  - 상세 Drawer의 `추천 관계 및 보상`은 관계 테이블로 렌더링하고, 피추천인 이름 클릭 시 `회원 상세`로 이동합니다.
  - 관계 테이블은 `진행 시각(가입/확정)`, `관계 상태`, `검수`, `누적 보상`을 기본 컬럼으로 표시합니다.
  - 각 관계의 보상 원장은 행 확장 영역의 하위 테이블로 확인합니다.
  - 하위 원장 테이블은 `유형`, `금액`, `처리 시각`, `사유` 중심으로 단순화해 좁은 Drawer 폭에서도 읽기 쉽게 유지합니다.
  - Drawer 본문 섹션 간 간격은 `32px`를 사용하고, 관계/원장/코드 단위 조정 테이블은 첫 번째 열 고정, 최대 5행 높이, 오른쪽 아래 페이지네이션 규칙을 적용합니다.
  - 관계에 귀속되지 않는 보상 조정은 `코드 단위 조정` 테이블에서 별도로 노출합니다.
  - 상세 Drawer에는 `기본 정보`, `추천 관계 및 보상`, `정책 스냅샷`, `이상치 및 운영 메모`가 구현되었습니다.
  - 파괴적 액션은 `ConfirmAction` 또는 조정 모달을 통해 사유 입력을 강제하고, 성공 후 `notification + AuditLogLink`를 제공합니다.

## 14. 오픈 이슈

- 추천 확정 시점이 정해지기 전에는 `추천 확정 수`, `보상 지급 건수`, `대기 상태`의 정의를 문서와 UI 모두 임시 표기로 관리해야 합니다.
- 보상 수단이 정해지기 전에는 `누적 보상` 컬럼의 포맷과 `커머스 > 포인트 관리` 연동 여부를 확정할 수 없습니다.
- 수동 보정 권한이 분리되지 않으면 보상 관련 고위험 액션이 과도하게 넓은 권한에 묶일 수 있습니다.
- 사용자 화면이 아직 없으므로, B2C 영향 표는 `확인됨`과 `운영상 추정`, `노출 예정`을 섞지 않고 상태값으로 분리 관리해야 합니다.
