# Commerce > 포인트 관리 상세 IA

## 1. 문서 목적

- 이 문서는 `/commerce/points` placeholder를 이후 실제 구현과 운영 협의의 기준 문서로 승격하기 위한 living IA다.
- 포인트는 추천인, 학습 미션, 이벤트, 결제/환불, 관리자 수동 조정이 동시에 연결되는 운영 데이터이므로, 정책이 미확정이어도 먼저 관리 포인트와 후속 결정 지점을 고정한다.
- 이 페이지는 구현 과정에서 운영/정책 논의에 따라 지속적으로 업데이트될 것을 전제로 하며, 확정값과 가정값을 혼합하지 않는다.
- 페이지 IA를 갱신할 때는 이 문서보다 먼저 `docs/guidelines/admin-ux-ui-design.md`, `docs/guidelines/admin-detail-drawer-guidelines.md`, `docs/specs/admin-page-tables.md`를 확인하고, 전역 일관성 기준 위에서 페이지 예외만 남긴다.

## 3. 페이지 목표와 비목표

### 목표

- 운영자는 포인트 정책, 포인트 원장, 소멸 예정 건을 한 화면에서 `검색 -> 상세 -> 조치 -> 감사 로그 확인` 흐름으로 관리해야 한다.
- 운영자는 추천인/미션/이벤트/결제/환불에서 발생한 포인트 변동을 사용자 단위 원장으로 교차 검수할 수 있어야 한다.
- 수동 적립, 수동 차감/회수, 소멸 보류 같은 고위험 액션은 반드시 사유와 근거를 남기고 후속 검증 링크를 제공해야 한다.
- 추후 B2C 포인트 지갑, 결제 포인트 사용, 리워드 내역의 SoT 후보를 문서 기준으로 먼저 정리한다.

### 비목표

- 외부 정산/회계/배치 시스템 전체를 이 화면이 직접 대체하지 않는다.
- 추천인, 미션, 이벤트, 결제/환불의 원본 정책을 이 페이지가 직접 소유하지 않는다. 이 페이지는 포인트 결과값과 원장 증적을 검수하는 허브 역할에 집중한다.
- 아직 확정되지 않은 적립/차감/소멸 정책을 최종 비즈니스 결정처럼 선언하지 않는다.

## 4. 운영자 사용 시나리오

- 시나리오 1: 운영자가 `정책` 탭에서 적립/차감/소멸 규칙을 검색하고, 활성 정책을 검수한 뒤 상태 변경 또는 수정 진입을 실행한다.
- 시나리오 2: 운영자가 `포인트 원장` 탭에서 특정 회원 또는 원천 이벤트를 검색하고, 수동 적립/차감/회수 내역을 상세 Drawer와 조정 Modal로 검증한다.
- 시나리오 3: 운영자가 `소멸 예정` 탭에서 임박 건을 검토하고 소멸 보류/해제 또는 내역 내보내기를 수행한 뒤 감사 로그와 원본 화면으로 후속 검수를 이어간다.

| 본문 목록 | 핵심 운영 데이터 비교와 대상 선택 | 정책 목록, 원장 목록, 소멸 예정 목록 | 행 클릭 상세, 상태 변경, 보류 등록 진입 | `Users`, `Content`, `Operation`, `Commerce`, `System`과 식별자 연결 | 잔액, 사용 가능 포인트, 소멸 예정 안내에 간접 영향 |
| 상세 Drawer/Modal | 세부 정보와 조치 근거 확인 | 정책 상세, 원장 상세, 소멸 계산 스냅샷, 수동 조정 입력, 보류 사유 입력 | 조회, 저장, 상태 변경, 감사 로그 확인 | 감사 로그/원본 화면과 직접 연결 | 포인트 노출 문구와 예외 처리 근거 |
| 후속 액션/보조 링크 | 원본 화면/감사 로그 이동 | `Target Type`, `Target ID`, 관련 회원/정책/이벤트 링크 | 감사 로그 확인, 원본 화면 이동 | 후속 검수 동선 고정 | 직접 영향 없음 |

## 6. 데이터 블록 정의

### 상단 요약 데이터

- `정책` 탭: `전체 정책`, `초안 정책`, `운영 중 정책`, `중지 정책` 카드를 사용한다. 카드는 현재 건수를 함께 보여주고 클릭 즉시 테이블을 필터링한다.
- `포인트 원장` 탭: `전체 원장`, `완료 원장`, `보류 원장`, `취소 원장` 카드를 사용한다. 카드 클릭은 `ledger.status`와 URL/store를 함께 갱신한다.
- `소멸 예정` 탭: `전체 소멸 예정`, `예정 건`, `보류 건`, `완료 건`, `취소 건` 카드를 사용한다. 카드 클릭은 `expiration.status`와 URL/store를 함께 갱신한다.

### 본문 상단 UI 데이터

- 메인 탭은 `정책`, `포인트 원장`, `소멸 예정` 3개를 사용한다.
- 메인 탭, 검색, 필터, 주요 액션 버튼은 모두 본문 컨테이너 안 `AdminListCard.toolbar`에 배치한다.
- `정책` 탭
  - 필터: `정책 상태`, `정책 유형`
  - 헤더 제어: `정책 유형`, `상태` 컬럼에 고정값 헤더 필터를 두고, `정책명`, `정책 유형`, `상태`, `최종 수정`은 헤더 정렬을 지원한다.
  - 검색: `정책명`, `정책 ID`
  - 우측 액션: `정책 등록`
  - `정책 등록` 버튼은 본문 상단 우측 끝에 두고 Ant Design `large` 크기를 사용한다.
- `포인트 원장` 탭
  - 필터: `원장 유형`, `발생 원천`, `처리 상태`, `기간`
  - 헤더 제어: `원장 유형`, `발생 원천`, `처리 상태` 컬럼에 고정값 헤더 필터를 두고, `발생 시각`, `원장 유형`, `발생 원천`, `포인트 증감`, `만료 예정일`, `처리 상태`는 헤더 정렬을 지원한다.
  - 검색: `회원 ID`, `회원명`, `원장 ID`
  - 우측 액션: `포인트 수동 조정`
  - `포인트 수동 조정` 버튼은 본문 상단 우측 끝에 두고 Ant Design `large` 크기를 사용한다.
- `소멸 예정` 탭
  - 필터: `소멸 상태`, `예정 기간`
  - 헤더 제어: `소멸 상태` 컬럼에 고정값 헤더 필터를 두고, `예정 일시`, `원천`, `예정 포인트`, `사용 가능 잔액`, `소멸 상태`는 헤더 정렬을 지원한다.
  - 검색: `회원 ID`, `회원명`, `소멸 ID`
  - 우측 액션: `소멸 보류 등록`, `내역 내보내기`
  - `소멸 보류 등록` 버튼은 본문 상단 우측 끝에 두고 Ant Design `large` 크기를 사용한다. `내역 내보내기`는 secondary action으로 같은 줄에 둔다.

### 상세 Drawer/Modal 데이터

- `정책 상세 Drawer`
  - `기본 정보`: 정책 ID, 정책명, 정책 유형, 상태, 최근 수정자/수정 시각
  - `적용 조건`: 대상 회원 조건, 트리거 이벤트, 중복 지급 허용 여부, 일/월/이벤트당 제한
  - `적립/차감 규칙`: 증감 수량, 지급/차감 시점, 회수 가능 여부, 음수 허용 여부
  - `소멸 규칙`: 기본 소멸 주기, 예외 대상, 사전 안내 기준
  - `운영 메모`: 검수 메모, 최근 변경 사유
- `포인트 원장 상세 Drawer`
  - `회원 요약`: 사용자 ID, 사용자명, 현재 잔액, 최근 원장 요약
  - `발생 정보`: 원장 ID, 원장 유형, 발생 원천, 증감 수량, 처리 시각, 처리 상태
  - `관련 정책/원본 이벤트`: 정책 ID/정책명, 추천/미션/이벤트/결제/환불 참조 링크
  - `사유/승인 정보`: 조정 사유, 승인자, 증빙 메모
  - `후속 링크`: 감사 로그, 회원 상세, 원본 정책/이벤트
- `소멸 예정 상세 Drawer`
  - `대상 요약`: 소멸 ID, 사용자, 예정 일시, 예정 포인트, 사용 가능 잔액
  - `소멸 계산 스냅샷`: 발생 원장, 남은 유효일, 계산 기준 정책
  - `보류/실행 이력`: 보류 여부, 보류 사유, 처리자, 처리 시각
  - `후속 링크`: 감사 로그, 원장 상세, 관련 정책
- `포인트 수동 조정 Modal`과 `소멸 보류 등록 Modal`
  - `Descriptions` 기반 입력 테이블을 기본값으로 사용한다.
  - 필수 항목 label(`th`)에는 빨간 `*`를 표시한다.
  - 사유/근거, 대상 식별자, 증감 포인트, 승인 메모, 첨부/링크 필드를 포함한다.
- `정책 등록/수정`
  - 현재 구현은 `Descriptions` 기반 입력 Modal이지만, 작성/편집 맥락이 강하므로 공통 UX 기준상 장기 방향은 전용 편집 페이지 또는 별도 편집 영역 우선이다.

| 소멸 보류 등록/해제 | `수정` | `CommercePointExpiration + expirationId` | 확인 + 사유 필수 | 보류 상태 반영 + 후속 로그 링크 표시 | `/system/audit-logs?targetType=CommercePointExpiration&targetId={expirationId}` |

| 포인트 발생 원천 분류 | `미확정` | `원장 유형`, `발생 원천`, 교차 검수 링크, 요약 카드 집계 기준이 달라진다. | 리워드 내역이 추천/미션/이벤트/결제 중 무엇으로 표기될지 달라진다. | `추천`, `미션`, `이벤트`, `결제`, `환불`, `관리자`, `시스템`을 기본 후보로 두고 최종 code table을 확정해야 한다. |
| 차감/회수 우선순위 | `미확정` | 어떤 잔액부터 먼저 차감하는지와 회수 시 잔액 계산 방식이 달라진다. | 사용자 지갑의 사용 가능 잔액, 소멸 예정 잔액이 달라진다. | `가장 먼저 만료되는 포인트 우선 차감` 여부와 프로모션성 포인트 우선 차감 여부를 정해야 한다. |
| 소멸 기준/사전 안내 | `미확정` | `소멸 예정` 탭 집계, 보류 큐 기준, 알림 배치 연결 규칙이 달라진다. | 소멸 예정 안내, 만료 문구, 보류 시 노출 정책이 달라진다. | 소멸 주기, 예외 대상, 사전 안내 시점, 소멸 보류 승인 규칙을 확정해야 한다. |
| 수동 조정 승인 체계 | `미확정` | 현재 `commerce.points.manage` 단일 권한으로 충분한지, 고액 조정 승인 분리가 필요한지 판단해야 한다. | 사용자 포인트가 운영자 승인 체계에 따라 수정될 수 있다. | 조정 한도, 2인 승인 여부, 증빙 첨부 의무를 정해야 한다. |
| 음수 잔액/복구 허용 여부 | `미확정` | 원장 상태값, 회수 실패 처리, 복구 액션 정책이 달라진다. | 사용자에게 표시되는 사용 가능 포인트와 예외 문구가 달라진다. | 음수 잔액 허용 여부, 실패 시 재시도/보정 정책을 정해야 한다. |
| URL/상태 복원 | `확정` | 탭/검색/상세 상태를 새로고침과 뒤로가기에서도 재현해야 한다. | 운영자는 같은 검수 맥락으로 복귀할 수 있다. | 탭별 쿼리 파라미터 네이밍을 구현과 맞춰 유지해야 한다. |
| 감사 추적 | `확정` | 조치가 있으면 `Target Type`, `Target ID`, 사유, 수행자 기준으로 감사 로그 확인 경로를 제공해야 한다. | 직접 B2C 노출이 없어도 운영 증적 확보가 필요하다. | 조회성 액션과 조치성 액션의 로깅 범위를 분리 관리해야 한다. |
| 정책 등록/수정 편집 방식 | `미확정` | 현재 구현은 Modal이지만, 문서 기준상 작성/편집 맥락이 강한 화면은 전용 편집 영역 또는 페이지가 우선이다. | 운영자가 정책을 안정적으로 조합해야 사용자 포인트 정책과 후속 노출 문구가 흔들리지 않는다. | `정책 등록/수정`을 페이지로 분리할지, 전용 편집 Drawer로 승격할지 결정해야 한다. |

- 현재 문서에서 권장하는 code table 초안
  - `policyStatus`: `초안`, `운영 중`, `중지`
  - `ledgerType`: `적립`, `차감`, `회수`, `복구`, `소멸`
  - `ledgerSourceType`: `추천`, `미션`, `이벤트`, `결제`, `환불`, `관리자`, `시스템`
  - `expirationStatus`: `예정`, `보류`, `완료`, `취소`

## 12. 네트워크 상태와 fail-safe

- `pending`
  - 상단 요약 카드와 현재 탭 목록에 skeleton/loading 상태를 노출한다.
  - 직전 성공 데이터가 있으면 화면 전체를 비우지 않고 유지한다.
- `success`
  - 탭별 결과를 정상 렌더링한다.
  - 고위험 액션 성공 시 `notification + AuditLogLink`를 함께 노출한다.
- `empty`
  - `정책`: `아직 등록된 포인트 정책이 없습니다`
  - `포인트 원장`: `조건에 맞는 포인트 원장이 없습니다`
  - `소멸 예정`: `현재 예정된 소멸 건이 없습니다`
- `error`
  - 탭 단위 에러 메시지, 재시도 버튼, 마지막 성공 상태 fallback 문구를 함께 노출한다.
  - 고위험 액션 실패 시 실패 사유와 재시도 가능 여부를 명확히 구분한다.
- 마지막 성공 상태 fallback
  - 상단 요약과 직전 탭 결과를 읽기 전용 상태로 유지한다.
  - Drawer는 대상 식별자만 유지하고 상세 재조회 실패 메시지를 별도 표시한다.
- 요청 취소/재시도
  - 탭 전환, 필터 변경, 화면 이탈 시 조회 요청을 abort한다.
  - 수동 조정과 소멸 보류 등록은 중복 제출 방지를 적용한다.
  - 내역 내보내기는 동일 조건 중복 실행 시 진행 중 상태를 분리 표시한다.

  - `PageTitle`

  - `AdminListCard`
  - `AdminDataTable`
  - `DetailDrawer`
  - `ConfirmAction`
  - `AuditLogLink`
- 예상 feature 파일
  - `src/features/commerce/pages/commerce-points-page.tsx`
  - `src/features/commerce/api/points-service.ts`
  - `src/features/commerce/model/point-store.ts`
  - `src/features/commerce/model/point-schema.ts`
- 상태 관리/데이터 소스 메모
  - 페이지 state는 `Zustand` 단일 원칙을 유지한다.
  - 탭/필터/query metadata는 `point-schema.ts` 같은 feature model 레이어에서 SoT로 관리한다.
  - 조회와 조치 모두 page-local 상수에 남기지 않고 service/store 경계로 분리한다.
- 권한/로그 처리 메모
  - 현재 기본 권한은 `commerce.points.manage`로 두되, 수동 조정/대량 소멸 보류는 추후 고위험 권한 분리를 검토한다.
  - 파괴적 액션에는 확인 단계, 사유 입력, `Target Type`, `Target ID`, 감사 로그 확인 경로를 반드시 함께 둔다.

## 14. 오픈 이슈

- 추천/미션/이벤트/결제/환불/관리자 조정 중 어떤 원천을 1차 code table로 확정할지 미정이다.
- 차감 우선순위와 소멸 계산 기준이 확정되지 않아, 사용자 잔액/예정 잔액 계산식이 바뀔 수 있다.
- 수동 조정 승인 체계와 증빙 첨부 의무가 아직 정해지지 않았다.
- 음수 잔액 허용 여부와 회수 실패 처리 규칙이 미정이다.
- 포인트 지갑, 결제 사용, 소멸 예정 안내의 실제 B2C 노출 surface는 구현 과정에서 추가 조정될 수 있다.
- 정책 등록/수정 UI는 현재 Modal이지만, 운영자가 규칙 조합을 길게 다루는 편집 흐름이므로 전용 페이지/편집 영역 전환 여부를 후속 결정해야 한다.

## 15. 운영/정책 업데이트 원칙

- 이 문서는 포인트 정책의 최종 확정본이 아니라, 구현과 운영 협의가 진행될수록 갱신되는 기준 문서다.
- 아래 항목이 바뀌면 같은 작업에서 이 문서와 관련 문서를 함께 갱신한다.
  - 탭 구조, 검색/필터, 컬럼, 액션
  - 엔티티명, 필드명, code table 후보
  - 감사 로그 `Target Type`, `Target ID`
  - B2C 노출 위치, 사용자 문구, 소멸 안내 방식
- 관련 동기화 문서
  - `docs/specs/admin-page-tables.md`
  - `docs/specs/admin-data-contract.md`
  - `docs/specs/admin-data-usage-map.md`
  - `docs/specs/admin-action-log.md`
  - `docs/specs/admin-page-gap-register.md`
- 정책이 미확정인 항목은 계속 `미확정`으로 표시하고, page-local 임시값을 최종 SoT처럼 굳히지 않는다.
