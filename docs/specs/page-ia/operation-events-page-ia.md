# Operation > 이벤트 상세 IA

## 1. 문서 목적

- 이벤트 화면의 운영 목적, 데이터 블록, 조치 흐름을 같은 기준으로 정리합니다.
- mock `service + Zustand store` 기반의 초기 구현 구조를 `목록 검수 + 등록/수정 상세 페이지` 혼합형으로 고정합니다.
- 운영 기본 흐름 `검색 -> 상세 -> 조치 -> 감사 로그 확인`과 편집형 화면의 `작성/수정 -> 저장/예약 -> 감사 로그 확인`을 한 화면군 안에서 연결합니다.

## 3. 페이지 목표와 비목표

### 목표

- 이벤트 목록을 검색하고 현재 노출 상태, 진행 기간, 보상 연결 상태를 빠르게 검수합니다.
- 이벤트 생성/수정은 별도 등록 상세 페이지에서 처리하고, 목록은 운영 검수와 조치에 집중합니다.
- 게시 예약, 종료 같은 조치 후 `Target Type`, `Target ID`, 감사 로그 확인 경로를 제공합니다.

### 비목표

- 백엔드 정산, 쿠폰 발급 엔진, 메시지 발송 엔진, 외부 광고 플랫폼 설정 전체를 이 화면이 직접 대체하지 않습니다.
- 이벤트 참여 현황의 원본 집계 로직을 이 화면이 소유하지 않고, 필요한 경우 원본 화면 또는 연결 패널로 이동합니다.
- 공개 이벤트가 아닌 내부 전용 이벤트까지 SEO 값을 강제 입력하지 않습니다. SEO/공유 메타는 공개 이벤트에 한해 선택적으로 override 합니다.

## 4. 운영자 시나리오

- 시나리오 1: 운영자가 SearchBar에서 이벤트명 또는 이벤트 ID로 검색하고 기간 필터를 적용해 대상 이벤트를 좁힙니다.
- 시나리오 2: 운영자가 목록 행을 클릭해 상세 Drawer에서 노출 상태, 참여 조건, 보상 정책, 메시지/쿠폰 연동 상태를 검수합니다.
- 시나리오 3: 운영자가 `이벤트 등록` 버튼 또는 Drawer/액션 셀의 `수정`으로 등록 상세 페이지에 진입해 이벤트 정보를 작성합니다.
- 시나리오 4: 운영자가 목록 또는 상세 Drawer에서 `게시 예약`, `종료`, `참여 현황 확인`을 수행하고 결과를 검수합니다.
- 시나리오 5: 운영자가 조치 후 `Target Type`, `Target ID` 기준으로 감사 로그를 확인하고 후속 화면 검수를 이어갑니다.

## 6. 데이터 블록 정의

### 상단 요약

- `노출 이벤트 수`
- `예약 이벤트 수`
- `숨김 이벤트 수`
- `전체 이벤트 수`는 SearchBar 우측 요약(`총 N건`)으로 유지하고, 상단 카드에는 중복 노출하지 않습니다.

### 본문 툴바

- SearchBar
  - 검색 대상: `이벤트명`, `이벤트 ID`
  - 검색어 입력
  - 상세 버튼
  - 총 건수
- `이벤트 등록` 버튼
  - 위치: SearchBar 우측 끝
  - 크기: `large`

### 본문 목록

- 이벤트 기본 정보
  - `이벤트 ID`
  - `이벤트명`
  - `유형`
  - `진행 기간`
  - `노출 상태`
- 진행 상태는 목록 컬럼에서 중복 노출하지 않고 `진행 기간`과 상세 Drawer에서 확인합니다.
- 운영 정보
  - `참여자 수`
  - `보상 정책 요약`
  - `최근 수정일`
  - `최근 수정자`

### 상세 Drawer

- 섹션 1. 기본 정보
  - 이벤트 ID, 이벤트명, 유형, 상태
- 섹션 2. 이벤트 요약
  - 이벤트 요약 텍스트
- 섹션 3. 이벤트 본문
  - 저장된 HTML 본문 미리보기
- 섹션 4. 노출 설정
  - 노출 채널, 공개 기간, 배너/랜딩 연결
- 섹션 5. 참여 조건
  - 대상 그룹, 참여 제한, 중복 참여 정책
- 섹션 6. 보상 정책
  - 보상 유형, 보상 정책, 쿠폰/포인트 연결
- 섹션 7. 메시지/쿠폰 연동
  - 안내 메시지 템플릿, 쿠폰 정책 링크
- 섹션 8. 운영 메모/변경 이력
  - 관리자 메모, 최근 수정 정보

### 등록 상세 페이지

- 섹션 표현 규칙
  - 등록 상세는 좌측 `Steps`(`progressDot`, `vertical`)로 섹션 순서를 항상 노출하고, 우측 본문에는 현재 선택된 step의 입력 패널만 노출합니다.
  - `Steps`는 `기본 정보 -> 이벤트 본문 -> 노출 설정 -> 참여 조건 -> 보상 설정 -> 노출/SEO 설정 -> 관리자 메모` 순서를 사용하며, 클릭 시 해당 step의 입력 항목으로 즉시 전환됩니다.
  - 저장 시 다른 step의 필수값 검증 오류가 발생하면, 첫 오류가 있는 step으로 자동 전환해 입력 복구 경로를 제공합니다.
- 섹션 1. 기본 정보
  - 이벤트명, 이벤트 유형, 상태
- 섹션 2. 이벤트 본문
  - `bodyHtml`
  - TinyMCE 기반 본문 편집기
  - 이벤트 상세/랜딩에 노출할 HTML 콘텐츠를 저장합니다.
- 섹션 3. 노출 설정
  - 노출 상태, 시작일/종료일, 노출 채널, 랜딩 URL, 배너
- 섹션 4. 참여 조건
  - 대상 그룹, 참여 제한, 안내 문구
- 섹션 5. 보상 설정
  - 보상 유형, 보상 정책, 지급 조건
- 섹션 6. 노출/SEO 설정
  - `slug`
  - `공유 제목`
  - `공유 설명`
  - `공유 이미지`
  - `canonical URL`
  - `index / noindex`
  - 기본값은 이벤트명/요약/배너로 자동 생성하고, 공개 이벤트만 선택적으로 override 합니다.
- 섹션 7. 관리자 메모
  - 운영 메모, 검수 메모

## 8. 상태값/정책/운영 규칙

| 항목 | 현재 상태 | 관리자 페이지 영향 | 사용자 화면 영향 | 추후 결정 필요 내용 |
| --- | --- | --- | --- | --- |
| 페이지 유형 | 설계 확정 | 목록은 검수와 조치, 등록 상세는 작성과 저장을 담당합니다. | 운영자가 같은 화면군에서 검색과 편집을 분리해 처리할 수 있습니다. | 실제 구현 시 상세 라우트와 브레드크럼 기준을 고정해야 합니다. |
| 상태값/운영 규칙 | 일부 미확정 | `진행 상태`, `노출 상태`, `예약 상태`, `종료 가능 조건`은 SoT 고정이 필요합니다. | 최종 상태 문구와 노출 정책이 함께 바뀔 수 있습니다. | 승인 체계, 종료 후 복구 가능 여부를 먼저 고정해야 합니다. |

| URL/상태 복원 | 확정 | 목록/정렬/선택된 Drawer 또는 등록 상세 진입 상태를 가능한 한 복원해야 합니다. | 운영자는 같은 검색/상세 맥락으로 복귀할 수 있습니다. | 필수 쿼리 파라미터를 변경하면 연관 화면도 함께 검토해야 합니다. |
| 감사 추적 | 확정 | 조치가 있으면 `Target Type`, `Target ID`, 사유, 수행자 기준으로 감사 로그 확인 경로를 제공합니다. | 직접 B2C 노출이 없어도 운영 증적 확보가 필요합니다. | 장기적으로 `OperationEvent` 단위 타깃 타입 분리 여부를 검토해야 합니다. |

  - `searchField`
  - `keyword`
  - `startDate`
  - `endDate`
  - `visibilityStatus`
  - `eventType`
  - `sortField`
  - `sortOrder`
  - `selected`
- Drawer 복원 여부: 권장
- 등록 상세 복원 여부: 권장
- 유지되어야 하는 상태: 목록 조건, 정렬, 선택된 상세 Drawer 상태, 등록 상세 진입 대상을 함께 복원하는 구조를 권장합니다.

## 12. 네트워크 상태와 fail-safe

- pending: 스켈레톤 또는 loading 상태를 표시하고, 직전 성공 데이터가 있으면 유지합니다.
- success: 목록, 상세 Drawer, 등록 상세를 정상 렌더링합니다.
- empty: 조건에 맞는 이벤트가 없음을 페이지 맥락에 맞게 명확히 안내합니다.
- error: 오류 코드/메시지, 재시도 버튼, 마지막 성공 상태 fallback 문구를 함께 노출합니다.
- 마지막 성공 상태 fallback: 화면 전체를 비우지 않고 직전 성공 데이터나 읽기 전용 요약을 유지합니다.
- 요청 취소/재시도: 화면 이탈 시 abort, 조회 실패 시 retry, 파괴적 액션은 중복 제출 방지가 필요합니다.
