# Message > 대상 그룹 상세 IA

## 1. 문서 목적

- 대상 그룹 화면의 운영 목적, 데이터 블록, 조치 흐름을 같은 기준으로 정리합니다.
- 구현 전 placeholder 화면은 미확정 정책과 후속 결정 포인트를 빈칸 없이 기록하고, 구현된 화면은 현재 코드/문서 기준 운영 흐름을 고정합니다.
- 운영 기본 흐름 검색 -> 상세 -> 조치 -> 감사 로그 확인 또는 편집형 화면의 작성/수정 -> 확인 -> 발행 -> 감사 로그 확인을 유지합니다.

## 3. 페이지 목표와 비목표

### 목표

- 대상 그룹 화면의 핵심 운영 데이터를 검색하고 검수합니다.
- 상세 확인과 조치 후 감사 로그 확인 경로를 같은 화면 문서 기준으로 고정합니다.

### 비목표

- 백엔드 정산, 배치, 외부 서비스 설정 전체를 이 화면이 직접 대체하지 않습니다.
- 연관 화면의 원본 책임을 빼앗지 않고 필요한 경우 원본 화면 이동과 감사 로그 확인 경로를 제공합니다.
- 이미 구현된 화면이라도 현재 코드/문서 SoT를 넘는 임의 규칙을 추가하지 않습니다.

## 4. 운영자 사용 시나리오

- 시나리오 1: 운영자가 대상 그룹 화면에서 검색/필터를 적용하고 대상 레코드나 편집 대상을 선택합니다.
- 시나리오 2: 운영자가 조회하기 흐름으로 세부 정보를 확인하고 필요한 조치를 실행합니다.
- 시나리오 3: 운영자가 조치 후 Target Type, Target ID 기준으로 감사 로그를 확인하고 관련 관리자 화면으로 후속 검수를 이어갑니다.

## 11. URL/상태 복원

- 기본 라우트: /messages/groups
- 필수 쿼리 파라미터 후보: searchField, keyword, startDate, endDate, selected, editor
- Drawer/Modal 복원 여부: 적용
- 유지되어야 하는 상태: 목록 조건과 선택된 그룹 상세 Drawer 상태를 함께 복원하고, 신규 생성 Drawer는 `editor=create`로 복원합니다.

## 12. 네트워크 상태와 fail-safe

- pending: loading 상태를 표시하고, 직전 성공 데이터가 있으면 같은 목록을 유지한 채 로딩만 중첩 표시합니다.
- success: 정상 결과를 렌더링합니다.
- empty: 조건에 맞는 데이터가 없음을 페이지 맥락에 맞게 명확히 안내합니다.
- error: 오류 코드/메시지, 재시도 버튼, 마지막 성공 상태 fallback 문구를 함께 노출하고 직전 성공 목록이 있으면 그대로 유지합니다.
- 마지막 성공 상태 fallback: 화면 전체를 비우지 않고 직전 성공 목록을 유지하며, fallback 가능 여부를 오류 문구로 함께 안내합니다.
- 요청 취소/재시도: 화면 이탈 시 abort, 조회 실패 시 retry, 파괴적 액션은 중복 제출 방지가 필요합니다.

## 13. 구현 메모

- 현재 코드베이스에서 재사용할 컴포넌트: PageTitle, SearchBar, AdminDataTable, ConfirmAction, AuditLogLink
- 예상 feature 파일: src/features/message/pages/*
- 본문 레이아웃 메모: `SearchBar`와 `그룹 추가` 버튼을 같은 `AdminListCard.toolbar` 줄에 두고, 안내 문구와 테이블은 카드 body 안에서 같은 기준으로 정렬합니다.
- 수정 진입은 별도 아이콘 없이 테이블 행 클릭으로 우측 Drawer를 열고, 작업 컬럼에는 `대상 수 재계산`, `그룹 삭제`만 둡니다.
- 그룹 등록/수정 Drawer의 기본 정보와 세그먼트 설정은 좌측 label / 우측 control 형태의 입력 테이블로 편집하고, `Form`은 검증과 상태 관리 용도로 사용합니다.
- 그룹 등록/수정 Drawer의 header/footer chrome은 공통 drawer frame을 사용하고, footer는 왼쪽 `취소`, 오른쪽 `조회하기`/`저장` `large` 버튼 묶음으로 정렬합니다.
- `조건 기반 그룹` Drawer는 `설정 유형`에 따라 `간편 설정` 또는 `상세 설정(Query Builder)` 본문을 즉시 교체하며, 신규 생성에서만 설정 유형을 변경할 수 있습니다.
- `상세 설정(Query Builder)`는 중첩 그룹/조건 추가와 `자연어/SQL/JSON` 변환 미리보기를 제공하고, 저장 시 SQL 문자열 요약과 query builder 구조를 함께 유지합니다.
- 세그먼트 옵션, 기본값, Query Builder 필드 정의는 `src/features/message/model/message-group-segment-schema.ts`에서 단일 SoT로 관리하고, page는 해당 스키마를 조합만 합니다.
- `상세 설정(Query Builder)`에는 실제 세그먼트 필드만 노출하고, `적용 채널`·`운영 상태` 같은 그룹 메타 필드는 포함하지 않습니다.
- `대상 수 재계산` 실행 중에는 해당 행의 `예상 대상 수` 숫자 위와 같은 그룹 Drawer의 `예상 발송 인원` 숫자 위에 `Spin size="small"` 로딩 표시를 오버레이로 노출하고, 숫자 opacity를 함께 낮춥니다.
- 목록 검색 조건은 URL에 유지하고, edit Drawer는 `selected={groupId}`, create Drawer는 `editor=create` 쿼리로 새로고침/뒤로가기 시 복원합니다.
- 그룹 목록 조회 실패 시 오류 Alert 안에 fallback 안내 문구를 노출하고, 직전 성공 목록이 있으면 테이블은 그대로 유지합니다.
- 목록 상태 메모: `상태` 컬럼은 `사용중/초안` 배지로 표시하고, 상태 변경은 등록/수정 Drawer 저장 흐름에서만 반영합니다.
- 권한/로그 처리 메모: 파괴적 액션에는 확인 단계와 사유 입력, Target Type, Target ID, 감사 로그 확인 경로를 함께 둡니다.

## 14. 오픈 이슈

- 실제 백엔드 세그먼트 DSL/SQL 계약이 확정되면 현재 프론트엔드의 query builder 구조(`queryBuilderConfig`)와 저장 포맷(`queryBuilderText`)을 서버 스키마 기준으로 다시 정렬해야 합니다.
