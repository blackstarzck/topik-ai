아래는 “프론트는 100% React로 만든다”를 전제로, 팀 규칙으로 바로 붙여 넣어도 되는 수준으로 **React 개발 최적화 룰**을 구체적으로 정의한 기준안이다. (관리자 화면처럼 테이블/필터/상세 이동이 많은 화면 구성에서는 특히 효과가 큼. )

---

## 0) 전제: “React 100% 사용”의 의미(금지/허용)

### 금지

* jQuery/DOM 직접 조작( `document.querySelector`, `innerHTML`, `addEventListener` 직접 붙이기 )

  * 예외: 반드시 필요한 경우 `ref` + `useEffect`로 래핑하고 “React 외부” 경계를 명확히 분리
* 전역 mutable singleton store를 임의로 만들어 컴포넌트 밖에서 상태 변경
* 컴포넌트 외부에서 DOM 상태를 신뢰(클래스 토글로 UI 상태 관리 등)

### 허용(단, 규칙 내에서)

* UI는 React 컴포넌트로만 구성
* DOM 접근은 `ref` 기반, 부작용은 `useEffect/useLayoutEffect`로만
* 상태는 “React state + 서버 상태 라이브러리(예: TanStack Query)” 중심으로

---

## 1) 컴포넌트 설계 규칙(렌더 최적화의 80%)

### 1-1. 컴포넌트 책임 분리(강제)

* Container/Presenter 분리

  * Container: 데이터 패칭/라우팅/상태 조합
  * Presenter: 순수 UI (props만으로 렌더)
* 한 컴포넌트의 “상태 종류”는 최대 3개(로컬 state 기준). 넘으면 쪼갠다.
* 리스트/테이블은 “Row 컴포넌트”를 반드시 분리한다(행 단위 리렌더 제한).

### 1-2. Props 안정성 규칙(강제)

* 부모에서 자식으로 내려가는 props는 **참조 안정성**을 유지해야 한다.

  * 함수: `useCallback`
  * 객체/배열: `useMemo` 또는 상수로 분리
* 렌더마다 새로 만들어지는 inline 객체/함수 props 금지

  * 금지 예: `<Row onClick={() => ...} style={{...}} filter={{...}} />`

### 1-3. key 규칙(강제)

* 리스트 key는 **절대 index 금지**
* 서버 ID 또는 안정적인 유니크 키만 사용

---

## 2) 상태 관리 규칙(로컬/서버/전역을 분리)

### 2-1. 상태 분류(필수)

1. **로컬 UI 상태**: 모달 열림, 탭, 인풋 값
   → `useState` / `useReducer`
2. **서버 상태**: 목록/상세/필터 결과, 캐시, 로딩, 재시도
   → TanStack Query(권장)로 통일
3. **전역 UI 상태**: 토스트, 전역 다이얼로그, 테마, 권한
   → 경량 store(zustand 등) 또는 Context(단, 아래 규칙 준수)

### 2-2. Context 사용 제한(성능 룰)

* “자주 바뀌는 값”을 Context에 넣지 않는다(예: 필터 값, 검색어, 테이블 정렬 상태)
* Context를 써야 한다면

  * Context를 기능 단위로 쪼갠다(하나의 거대한 AppContext 금지)
  * Provider 안에서 value 객체는 `useMemo`로 고정

### 2-3. derived state 금지

* props/서버 데이터로 계산 가능한 값을 `useState`로 “복사” 금지
* 계산 값은 `useMemo` 또는 렌더 내부 계산(가벼울 때)로 유지

---

## 3) 렌더링/메모이제이션 룰(언제 쓰고, 언제 쓰지 말지)

### 3-1. React.memo 적용 기준(강제 기준)

다음 중 하나라도 해당하면 `React.memo` 적용:

* 테이블 Row/Cell 같이 **개수가 많은 반복 컴포넌트**
* props가 무겁고(객체/배열) 리렌더 비용이 큰 컴포넌트
* 부모가 자주 리렌더되는 구조에서 자식은 변동이 적은 경우

적용하지 말아야 하는 경우:

* props가 거의 매번 바뀌는 컴포넌트(메모이제이션 비용만 증가)

### 3-2. useMemo/useCallback 적용 기준(강제 기준)

* `useMemo`

  * 정렬/필터링/그룹핑 등 O(n) 이상 계산이 렌더마다 발생할 때
  * 테이블 columns 정의/row 렌더러 등 “참조가 바뀌면 전체 리렌더”를 유발하는 객체
* `useCallback`

  * Row/Item에 내려가는 핸들러
  * dependency가 3개 이상이면 로직을 분리(커스텀 훅 또는 유틸)

### 3-3. effect 규칙(강제)

* `useEffect`에서 state set이 발생하면 “의도된 재실행 조건”을 코드로 설명 가능해야 함
* 의존성 배열 생략 금지(정당한 이유 없으면)
* 구독/타이머/이벤트는 반드시 cleanup 작성

## 4) 대용량 목록/테이블 최적화(관리자 페이지 핵심)

관리자 테이블(회원 목록/게시글/결제 등)처럼 행이 많으면 아래를 기본값으로 둔다. 

### 4-1. Virtualization 필수 기준

* 한 화면에서 200행 이상 가능성이 있으면 **무조건 virtualization**(react-window/react-virtual 등)
* 셀 렌더에 이미지/뱃지/버튼 등 복합 UI가 있으면 100행부터 적용 권장

### 4-2. 페이지네이션/필터링은 서버 기준

* 클라이언트에서 전체 데이터를 받아 필터링/정렬 금지(데이터 규모 불명확하면 특히)
* 서버 쿼리 키 기준(예: `['users', {page, pageSize, sort, filters}]`)으로 캐싱

### 4-3. 입력 디바운스 규칙

* 검색 input은 300ms 디바운스(기본)
* 즉시 반영이 필요한 필터(checkbox)는 즉시 요청 가능, 단 “연속 변경”은 배치

---

## 5) 라우팅/코드 스플리팅/번들 규칙

### 5-1. 라우트 단위 lazy 로딩(강제)

* “페이지(라우트)”는 모두 `React.lazy` 또는 프레임워크 표준(next/dynamic 등)으로 분리
* 모달도 큰 폼/에디터면 lazy 로딩

### 5-2. 프리패치 규칙

* 목록 → 상세 이동이 잦으면

  * Row hover/viewport 진입 시 “상세 프리패치” (TanStack Query `prefetchQuery`)
* 단, 프리패치 남발 금지: 최근 N개(예: 10개)까지만

### 5-3. 번들 경고 기준(팀 룰)

* 초기 로드 JS가 커지는 변경은 PR에서 “증분 용량” 코멘트 필수
* 공통 유틸로 빼기 전: 트리쉐이킹 가능한지 확인(통으로 import 금지)

---

## 6) 네트워크/서버 상태(캐시/리트라이/동기화)

### 6-1. 서버 상태 표준

* 목록/상세는 TanStack Query 기준
* `staleTime` / `gcTime` 기본 정책을 프로젝트 전역으로 통일
* mutation 성공 후 동기화:

  * 가장 가까운 query만 invalidate(전역 invalidate 남발 금지)
  * 가능하면 optimistic update + 롤백

### 6-2. 로딩 UX 규칙

* 페이지 로딩: skeleton 사용
* 테이블 내부 로딩: “이전 데이터 유지 + 상단 로딩 표시”(keepPreviousData 계열)
* 에러: “재시도 버튼 + 오류 코드/메시지” 표준 컴포넌트로 통일

---

## 7) React 18 동시성 기능 사용 규칙(필요할 때만)

* `useTransition`

  * 검색/필터 변경처럼 UI 반응성 유지가 중요한 경우에만
* `useDeferredValue`

  * 입력값에 따라 큰 리스트가 리렌더될 때만
* 남용 금지: 디바운스/가상화가 1순위

---

## 8) 폴더/코드 구조 규칙(성능과 유지보수 함께 잡기)

### 8-1. Feature 기반 구조(권장)

* `features/users/*`, `features/community/*` 처럼 “도메인 단위”로 묶고
* 각 feature 안에 `components/`, `hooks/`, `api/`, `types/` 분리
* 공통은 `shared/`로 이동(최소화)

### 8-2. 커스텀 훅 규칙

* 훅은 “하나의 책임”만
* 훅 내부에서 UI를 렌더하지 않는다(컴포넌트 반환 금지)

---

## 9) Lint/리뷰에서 강제할 체크리스트(실행 규칙)

PR 리뷰에서 아래 항목은 “위반 시 수정 후 머지”:

1. 리스트 key에 index 사용 금지
2. inline 객체/함수 props 남발 금지(반복 컴포넌트에서 특히)
3. 대용량 테이블에 virtualization 미적용(기준 충족 시)
4. derived state 복사 금지
5. effect 의존성 누락 금지
6. 서버 상태를 로컬 state로 중복 관리 금지
7. 라우트 코드 스플리팅 누락 금지(페이지 단위)

---

## 10) 성능 목표(팀이 측정 가능한 기준)

* 테이블 스크롤 시 “프레임 드랍이 체감될 정도”면 실패로 간주하고 최적화 작업 진행
* 입력(검색/필터) 변경 후 UI 반응: 즉시(로딩 표시라도 바로) 나타나야 함
* 동일 화면에서 불필요한 리렌더가 반복되면(예: Row 전체가 매번 리렌더) 구조 개선 우선
