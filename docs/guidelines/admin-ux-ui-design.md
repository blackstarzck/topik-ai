# TOPIK AI Admin UX/UI 원칙

## 문서 계약

- 기준 모듈: Dashboard, Users, Community, Message, Operation, Commerce, Assessment, Content, Analytics, System
- 메뉴명은 `Users`를 코드 식별자로 유지하되, 사용자 노출 라벨은 역할에 맞는 한글 명칭을 사용합니다.
- 감사 추적 용어는 `감사 로그`로 통일합니다.
- Users 상세 탭은 `프로필`, `활동`, `결제`, `커뮤니티`, `로그`, `관리자 메모` 6개를 고정합니다.
- 사용자가 `~을 참고해서`, `~와 비슷하게`, `~처럼 맞춰서`처럼 레퍼런스 기반 표현을 쓰더라도, 명시적 예외 요구가 없는 한 기존 전역 공통 레이아웃/공용 컴포넌트/일관성 규칙을 우선합니다.
- 페이지 IA와 개별 페이지 문서는 shared 레이아웃/스타일의 SoT가 아니며, 전역 공통 문서에 이미 있는 baseline을 다시 정의하지 않습니다.
- `Drawer`/`Modal` 폭, gap, padding, `Descriptions` label width, 공통 버튼 크기처럼 shared 책임인 수치는 페이지 문서에 반복 기재하지 않고, 예외가 있을 때만 별도 `UI 예외` 섹션에 남깁니다.

## 라벨 언어 규칙

- 사용자에게 노출되는 메뉴, 브레드크럼, 페이지 제목, 탭, 버튼, 상태 라벨은 기본 한글로 작성합니다.
- 동일 개념을 영어와 한글로 혼용하지 않습니다.
- 영문 표기 예외는 제품명, 외부 서비스명, 코드/식별자, 법적 고유 명칭처럼 한글 치환 시 의미가 손상되는 경우로 제한합니다.
- 코드 식별자와 URL 경로는 영문을 유지할 수 있지만, 화면 라벨은 별도 한글 매핑을 둡니다.

## 운영 기본 흐름

- 공통 흐름은 `검색 -> 상세 -> 조치 -> 감사 로그 확인`입니다.
- 콘텐츠형 화면은 `작성/수정 -> 확인 -> 발행 -> 감사 로그 확인` 흐름을 추가로 가집니다.
- 조치성 화면은 성공 피드백과 함께 `감사 로그 확인` 진입점을 제공해야 합니다.

## 관리자 페이지 유형

관리자 페이지는 아래 4가지 패턴으로 분류합니다. 새 페이지를 만들 때는 이 패턴 중 하나를 먼저 선택하고, 예외가 필요할 때만 변형합니다.

### 1) 목록 운영형

- 목적: 대량 데이터 조회, 필터링, 상태 조치
- 기본 구성:
  - `PageTitle`
  - 필요 시 요약 카드 3~4개
  - 목록형 요약 카드는 shared `ListSummaryCards`를 기본 컴포넌트로 사용
  - `AdminListCard` 또는 같은 구조의 공용 본문 카드
  - 카드 `toolbar` 안의 페이지 단위 UI 묶음
- `SearchBar`를 쓰는 페이지는 `검색 대상 Select + 검색 Input + 상세 버튼 + 총 건수`를 유지하고, `등록/추가/생성` 버튼은 같은 줄 우측 끝에 둡니다.
- 실제 검색 대상 필드가 1개뿐인 페이지도 공통 목록형 레이아웃 일관성을 위해 좌측 선택기를 숨기지 않고 `전체` 1옵션으로 유지할 수 있습니다.
  - 탭이 필요한 페이지는 카드 header `extra`나 본문 밖 별도 줄이 아니라 카드 `toolbar` 안에서 `SearchBar` 위쪽 행으로 정렬합니다.
  - `SearchBar`가 없는 페이지는 본문 상단 우측 끝에 `등록/추가/생성` 버튼을 두고 그 아래에 테이블 또는 주요 데이터 블록을 둡니다.
  - 상세 조건은 `상세` 버튼 클릭 시 열리는 드롭다운 패널로 분리하고, 페이지 기준 날짜 필드의 기간 범위만 배치합니다. 패널 하단에는 `필터 초기화`와 `적용` 버튼을 둡니다
- 드롭다운 안에서 바꾼 기간 값은 즉시 조회하지 않고 draft 상태로만 유지합니다. 실제 목록 조회는 `적용` 버튼 클릭 시점에만 발생합니다.
  - `상세 검색` 드롭다운에는 항상 `DatePicker.RangePicker` 기반 기간 범위 필터를 포함합니다
  - 메인 `Table`
  - 행 클릭 `Drawer` 또는 상세 모달
- 추천 컴포넌트:
  - `Card`, `ListSummaryCards`, `Select`, `Input`, `Modal`, `DatePicker`, `Table`, `Drawer`, `Tag`, `Popconfirm`
- 적합한 페이지:
  - 회원 목록
  - 강사 관리
  - 추천인 관리
  - 쿠폰 관리
  - 포인트 관리
  - 이커머스 관리

### 2) 정책/시나리오 편집형

- 목적: 정책, 규칙, 시나리오, 운영 설정 편집
- 기본 구성:
  - `PageTitle`
  - 상단 상태/버전 바
  - 좌측 목록 또는 상단 탭
  - 우측 편집 폼 또는 메인 편집 영역
  - 저장/게시/복제 액션 바
  - 테스트 또는 변경 이력 패널
- 등록 상세 서브패턴:
  - `Operation > 이벤트 등록 상세`, `Commerce > 쿠폰 등록 상세`처럼 단계형 등록 상세는 `PageTitle -> AdminListCard(toolbar actions) -> 좌측 Steps(progressDot, vertical) + 우측 현재 step 입력 패널` 구조를 공통 baseline으로 사용합니다.
  - 같은 패턴의 등록 상세는 shared `AdminEditorForm` shell을 사용하고, 도메인 안내 Alert나 placeholder note는 shell 바깥의 별도 hero로 빼지 않고 현재 step 섹션 내부에서만 노출합니다.
- 추천 컴포넌트:
  - `Form`, `Tabs`, `Segmented`, `Card`, `Alert`, `Drawer`, `Modal`, `Timeline`
- 적합한 페이지:
  - 이벤트
  - 쿠폰 등록 상세
  - 챗봇 설정
  - 학습 미션

### 3) 카탈로그/자산 관리형

- 목적: 자산 목록과 메타데이터 관리
- 기본 구성:
  - `PageTitle`
  - 유형/상태 중심 필터
  - 메인 `Table`
  - 우측 상세 `Drawer`
  - 상세 안 탭: 기본 정보, 메타데이터, 연결 자산, 변경 이력
- 추천 컴포넌트:
  - `Table`, `Drawer`, `Descriptions`, `Tabs`, `Tag`, `Badge`, `Empty`
- 적합한 페이지:
  - 문제은행
  - 콘텐츠 관리
  - 배지
  - 단어장

### 4) 편집/미리보기 병행형

- 목적: 콘텐츠/시험 구성과 결과 미리보기를 동시에 확인
- 기본 구성:
  - `PageTitle`
  - 상단 상태/버전/발행 여부
  - 좌측 편집 폼
  - 우측 미리보기 패널
  - 하단 저장/검수/발행 액션
- 추천 컴포넌트:
  - `Form`, `Card`, `Splitter` 또는 2열 레이아웃, `Tabs`, `Preview Panel`, `Affix` 액션 바
- 적합한 페이지:
  - EPS TOPIK
  - 레벨 테스트
  - 소나기
  - 객관식 선택

## 레이아웃 배치 규칙

- `PageTitle`/브레드크럼과 overlay(`Drawer`, `Modal`, `Popover`, `Tooltip`)를 제외한 페이지 단위 UI는 반드시 본문 컨테이너 안에 둡니다.
- 탭, 필터, 검색, 요약 액션, 주요 버튼을 본문 밖 별도 줄이나 카드 헤더 `extra`에 분리하지 않습니다.
- 목록형 페이지: `PageTitle -> Summary Cards(선택) -> AdminListCard(toolbar=본문 상단 UI, body=안내 문구 -> Table)`
- 목록 운영형 페이지의 요약 카드는 shared `ListSummaryCards`를 기본값으로 사용하고, page-local `Row + Card + Statistic` 조합을 새로 만들지 않습니다.
- `main` 안에서 `PageTitle` 아래에 오는 공용 본문 카드(`AdminListCard`, placeholder용 공용 카드)는 남는 세로 공간을 최소 높이로 채워야 합니다. 카드 배경이 `main` 하단 전에 끊기지 않도록 유지하고, 내부 콘텐츠가 적더라도 본문 영역 전체가 비어 보이지 않게 맞춥니다.
- 목록형 페이지에서 가로 스크롤이 있는 공용 테이블(`AdminDataTable`)을 본문 카드 안에 둘 때는, 테이블 viewport와 하단 스크롤 영역도 카드 하단까지 이어지게 유지합니다. 행 수가 적더라도 스크롤바가 중간에서 끊겨 보이지 않게 맞춥니다.
- 가로 스크롤이 필요한 경우에도 테이블 wrapper, pagination, 하단 스크롤 영역은 카드 본문 너비 안에 남아 있어야 합니다. 초과 폭은 카드 바깥으로 밀어내지 말고 테이블 내부 스크롤에서만 처리합니다.
- 1depth 목록 페이지가 첫 조회 `pending` 상태에 들어가면 테이블을 내리지 말고 shared `AdminDataTable`을 그대로 유지합니다. 초기 데이터가 아직 없을 때는 테이블의 empty 상태 위에 loading 오버레이가 겹치는 구조를 기본값으로 사용합니다.
- 1depth 목록의 첫 조회 `pending` 상태를 본문 `Alert` 한 줄로 대체하지 않습니다. `불러오는 중입니다`류의 안내 `Alert`는 마지막 성공 데이터가 남아 있는 재조회 상황에서만 선택적으로 사용하고, 이때도 테이블은 계속 보여야 합니다.
- `등록/추가/생성/새로 만들기` 계열 버튼은 본문 기준 테이블 또는 주요 데이터 블록의 우측 상단에 둡니다.
- 본문 상단의 `등록/추가/생성/새로 만들기` 계열 버튼은 항상 Ant Design `large` 크기를 사용합니다.
- `SearchBar`가 있는 목록형 페이지에서는 `SearchBar`와 `등록/추가/생성` 버튼을 같은 줄에 두고, 버튼은 항상 우측 끝에 정렬합니다.
- `SearchBar.actions`에 배치되는 우측 액션 버튼은 `등록/추가/생성` 여부와 관계없이 모두 `large` 크기로 통일합니다. `내보내기`, `새로고침`, `설정`, `즉시 실행` 같은 보조 액션도 예외를 두지 않습니다.
- `SearchBar`의 총 건수와 `내보내기` 같은 핵심 보조 액션도 가능하면 같은 줄 오른쪽 영역에 두고, 카드 header `extra`로 따로 빼지 않습니다.
- 탭이 있는 목록형 페이지에서도 우측 액션을 탭 줄에 따로 묶는 전용 헤더를 만들지 않습니다. 기본 구조는 `Tabs -> SearchBar(summary + actions)` 두 줄이며, 버튼/총 건수는 `SearchBar.actions`와 `SearchBar.summary` 슬롯을 사용합니다.
- 탭과 `SearchBar` 사이 간격은 공용 `AdminListCard.toolbar`의 세로 간격을 그대로 사용합니다. 페이지별 absolute 배치나 전용 wrapper로 탭 아래 여백을 따로 보정하지 않습니다.
- 탭과 우측 액션 버튼을 같은 행에 둘 때도 탭 하단의 회색 baseline은 부모 영역 전체 너비로 이어져야 합니다. 탭 nav 폭만큼만 끊겨 보이게 두지 않으며, 선은 헤더 블록 맨 아래가 아니라 탭 바로 하단에 붙어 보여야 합니다.
- `SearchBar`가 없는 목록형 페이지에서는 본문 상단 우측 끝에 `등록/추가/생성` 버튼을 두고, 버튼 아래에 안내 문구와 테이블을 배치합니다.
- 우측 상단 버튼은 `등록/생성`, `내보내기`, `즉시 실행`처럼 해당 페이지의 핵심 액션이 분명할 때만 둡니다. 단순 페이지 이동 버튼은 안내 문구나 테이블 링크로 내립니다.
- `SearchBar` 바깥에는 검색과 직접 관련 없는 필터를 두지 않는 것을 기본값으로 삼되, `전체/상태/유형`처럼 현재 목록 범위를 빠르게 전환하는 단일 선택 필터는 `PageTitle` 아래 요약 카드 행으로 분리할 수 있습니다.
- `정렬`, `상태`, `카테고리`, `역할`, `수단`처럼 비기간 조건은 `상세 검색`에 넣지 않습니다.
- 테이블 컬럼 필터는 코드값, 정책값, 상태값처럼 운영 기준이 이미 정해진 고정값 집합 컬럼에만 둡니다.
- 제목, 이름, ID, 자유 텍스트, 숫자, 금액, 건수, 비율, 날짜/시각 컬럼은 컬럼 필터를 두지 않고 정렬과 상단 검색/기간 필터로만 제어합니다.
- 상태 컬럼 필터도 고정된 상태 세트가 있고 운영상 좁혀볼 목적이 분명할 때만 유지하며, URL/store 동기화가 걸린 제어 필터는 반드시 유지합니다.
- 관리자가 목록에서 직접 전환할 수 있는 2상태 정책값은 별도 `상태`/`발행 상태` 컬럼으로 분리하고, `BinaryStatusSwitch` 같은 switch 컴포넌트로 조치합니다. 같은 조치를 `더보기` 메뉴에 중복 노출하지 않습니다.
- 특정 유형에서만 제어 가능한 정책값이라면 비대상 행에 disabled switch를 남기지 않고 `해당 없음`처럼 정책 비적용 상태를 명시합니다.
- `전체/상태/유형`처럼 현재 목록 범위를 빠르게 전환하는 단일 선택 필터는 본문 `SearchBar`의 `extra` 영역보다 `PageTitle` 아래 요약 카드 행을 우선 사용합니다. 카드는 현재 건수를 함께 보여주고 클릭 시 테이블/목록이 즉시 필터링돼야 합니다.
- 클릭 가능한 요약 카드는 현재 선택 상태를 지속 테두리나 아웃라인으로 강조하지 않습니다. hover에서도 카드 외곽선/그림자/이동 효과를 주지 않고, 클릭 순간의 ripple 모션과 결과 목록 변화로만 피드백합니다. 키보드 탐색 시에만 `focus-visible` outline을 유지합니다.
- `SearchBar.extra`에 체크박스 그룹을 나란히 두는 패턴은 목록형 페이지의 기본 레이아웃으로 사용하지 않습니다. 검색 입력부와 직접 관련된 보조 정보가 아닌 이상, 범위 전환 필터는 요약 카드로 승격합니다.
- 동일한 목록형이라면 `검색 대상 -> 검색어 -> 상세 -> 총 건수` 위치를 페이지마다 바꾸지 않습니다. `메시지 > 메일/푸시`, `커뮤니티 > 게시글 관리/신고 관리`, `운영 > FAQ`는 동일한 카드 내부 목록 패턴을 기준으로 맞춥니다.
- 상세 drill-down: 링크 이동 또는 `Drawer`
- 사용자 관련 클릭 이벤트에서 `사용자명`을 링크로 노출할 때는 공통 `table-navigation-link` 스타일(파란색 링크 텍스트)을 사용합니다.
- 사용자명 클릭 링크는 `회원 상세`로 이동하는 단일 동선으로 유지하고, 같은 목적의 중복 액션 버튼이나 `회원 상세` 텍스트 링크를 별도로 두지 않습니다.
- `table-navigation-link`와 파란 링크 텍스트는 실제 라우트 이동이 발생하는 셀에만 사용합니다. Drawer 열기, 상세 선택, 복사, 정지, 미리보기, 드롭다운 열기처럼 페이지 전환이 없는 셀 액션에는 절대 사용하지 않습니다.
- 페이지 전환이 없는 셀 액션은 행 클릭, `text/default` 버튼, `더보기` 드롭다운 중 하나로 처리하고, 링크처럼 보이는 시각 스타일을 섞지 않습니다.
- 편집형 페이지: `PageTitle -> Status Bar -> Editor -> Action Bar`
- 단계형 등록 상세 페이지: `PageTitle -> AdminListCard(toolbar actions) -> AdminEditorForm(좌측 Steps + 우측 현재 section)` 구조를 유지하고, 이벤트/쿠폰처럼 같은 등록 패턴끼리는 같은 shell과 spacing을 재사용합니다.
- placeholder 페이지: `PageTitle -> 정보 Alert -> 권한 안내 -> 예정 기능 리스트`

## 컴포넌트 선택 규칙

- 필드 또는 컬럼 설명을 추가해야 할 때는 heading/label 오른쪽에 정보 아이콘(`i`)을 배치하고, 마우스 hover 또는 키보드 focus 시 `Tooltip`으로 설명 메시지를 노출합니다. 설명 문구를 항목 하단 helper text로 상시 노출하는 패턴은 기본값으로 사용하지 않습니다.
- 목록의 빠른 조치는 `Table` 행 단위 액션으로 제공합니다.
- 테이블 액션 컬럼은 shared `TableActionMenu`의 `더보기` 버튼을 기본값으로 사용합니다. `액션 + 아이콘` 같은 페이지 전용 버튼 조합은 공용 패턴이 이미 있을 때 새로 만들지 않습니다.
- 목록형 테이블에 `액션` 컬럼이 있으면 shared `AdminDataTable` 기준으로 우측 고정(`fixed: 'right'`)을 기본값으로 사용합니다.
- 행 클릭으로 상세 Drawer를 여는 테이블에서는 `더보기`, 스위치, 셀 내부 버튼/보조 액션이 행 클릭으로 버블링되지 않게 반드시 `stopPropagation`을 적용합니다.
- shared `TableActionMenu`에서 `정지/중지/삭제/숨김/회수`처럼 운영상 중요한 액션은 메뉴 본문과 footer 영역을 분리하고, footer 상단 구분선과 강조 배경을 기본값으로 사용합니다.
- 운영상 중요한 액션은 `danger: true`를 우선 명시하고, footer 버튼은 full-width 강조 버튼으로 노출합니다. 해제/재개/복구처럼 되돌리는 액션은 같은 footer 규칙으로 올리지 않습니다.
- shared `AdminDataTable`을 쓰는 목록형 테이블의 header/body 셀과 셀 내부 요소는 왼쪽 정렬을 기본값으로 유지합니다. 페이지별 컬럼 정의에서도 별도 `align` 예외를 두지 않고 같은 기준을 따릅니다.
- shared `AdminDataTable`의 기본 렌더링 모드는 native `table / thead / tbody / tr / th / td` DOM을 유지합니다. 페이지 단위에서 raw Ant Design `virtual` prop으로 DOM 타입을 바꾸지 않습니다.
- 상세 확인은 전체 페이지 이동보다 `Drawer`를 우선 검토합니다.
- 행 클릭 상세 Drawer의 헤더/본문/푸터 구조는 `docs/guidelines/admin-detail-drawer-guidelines.md`를 단일 기준으로 따릅니다.
- `Drawer` 내부에서 반복 데이터 컬렉션을 보여줄 때는 `List`보다 `Table`을 우선 사용합니다.
- `Drawer` 내부 테이블에 긴 본문(예: 관리자 메모, 운영 코멘트)이 포함되면 셀에서는 요약만 보여주고, 전체 내용은 expandable row에서 확인하게 구성합니다.
- 행 클릭 상세 Drawer의 헤더 오른쪽은 상태/태그 같은 읽기 전용 메타만 두고, 전역 조치 버튼은 푸터 오른쪽으로 내립니다.
- 행 클릭 상세 Drawer의 푸터 왼쪽은 `감사 로그 확인`, 오른쪽은 주요 조치 버튼 묶음을 기본 위치로 사용합니다.
- 행 클릭 상세 Drawer의 header/footer 시각 baseline은 `Community > 게시글 관리 > 게시글 상세 Drawer`를 따르고, header 메타 간격 `8px`, footer 버튼 간격 `8px`, footer wrap 구조를 기본값으로 유지합니다.
- 행 클릭 상세 Drawer footer 액션 버튼은 공통 `large` 크기를 기본값으로 사용합니다.
- 행 클릭 상세 Drawer는 마지막 본문 블록과 footer 사이에 최소 `16px` 여백을 둬 footer 구분선과 본문 카드가 바로 맞닿지 않게 유지합니다.
- 작성/편집/조건 설정 Drawer도 본문 구조는 화면 목적에 맞게 다를 수 있지만, header/footer chrome은 같은 baseline을 따릅니다.
- 작성/편집/조건 설정 Drawer footer는 왼쪽 `취소`, 오른쪽 보조/주요 액션 버튼 묶음, 버튼 `large`, gap `8px`, wrap 허용 구조를 기본값으로 사용합니다.
- `Drawer` 내부 섹션 간 간격은 `Space` 기준 `32px`를 기본값으로 사용합니다.
- `Drawer` 내부 최상위 테이블은 첫 번째 열을 좌측 고정하고, 본문 높이는 최대 5개 행 정도가 보이도록 유지합니다.
- `Drawer` 내부 최상위 테이블 페이지네이션은 테이블 기준 오른쪽 아래(`bottomRight`)에 배치합니다.
- `Drawer` 내부 expandable row에 렌더되는 하위 테이블은 첫 번째 열을 고정하지 않고, 페이지네이션 없이 한 번에 표시합니다.
- `DetailDrawer` 내부 `Descriptions`의 label(`th`) 폭은 전역 `130px`로 통일하고, 페이지별 width override는 예외 사유가 있을 때만 허용합니다.
- 위 레이아웃 규칙이 바뀌면 `docs/guidelines/admin-detail-drawer-guidelines.md`, `docs/specs/admin-page-tables.md`, 영향받는 `docs/specs/page-ia/*.md`를 같은 작업에서 갱신합니다.
- 작성/편집 맥락이 강한 화면은 `Modal`보다 별도 편집 영역 또는 전용 페이지를 우선 사용합니다.
- 정책 충돌이나 파괴적 액션은 `ConfirmAction` 또는 `Modal.confirm`으로 명시적으로 확인합니다.
- 입력 Modal/Drawer/보조 패널의 항목형 UI는 일반 `Form.Item` 세로 나열보다 shared `AdminFormDescriptions` 또는 같은 `Descriptions` wrapper를 기본값으로 사용합니다. `Form`은 검증과 상태 관리 용도로만 유지하고, 단일 본문 에디터 같은 캔버스형 입력만 예외로 둡니다.
- `Descriptions` 기반 입력 테이블에서 필수 검증이 있는 항목은 label(`th`) heading에 빨간 `*`를 붙여 필수 입력 여부를 시각적으로 고정합니다.
- 입력형 `Descriptions` baseline을 페이지별 plain `Form.Item` 세로 나열로 되돌려서는 안 되며, 예외가 필요하면 페이지 IA `UI 예외` 섹션과 구현 메모에 이유/영향 범위를 함께 남깁니다.

## 상호작용 규칙

- 파괴적 액션은 확인 단계와 사유 입력 수단을 제공합니다.
- 성공 피드백은 대상 요약과 `감사 로그 확인` 경로를 함께 제공합니다.
- URL 파라미터로 목록, 필터, 탭 상태를 복원합니다.
- Drawer나 편집 패널을 닫아도 목록 필터와 선택 상태는 가능한 한 유지합니다.

## 사용자 표시 규칙

- 테이블에서 사용자 또는 사용자 계정 참조 셀은 raw ID를 단독으로 노출하지 않고 `이름 (ID)` 형식의 공통 `table-navigation-link`로 표시합니다.
- 사용자명 링크는 항상 `Users > 회원 상세`로 이동하는 단일 동선으로 유지하고, 상세 탭이 정해진 경우에만 해당 탭 쿼리를 함께 전달합니다.
- Drawer, Modal, Descriptions 등 상세 정보 영역에서 사용자명이 들어가는 필드는 `이름 (ID)` 형식을 유지합니다.
