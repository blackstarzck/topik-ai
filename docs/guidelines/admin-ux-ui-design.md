## UX 원칙

- 운영 기본축은 `검색 -> 상세 -> 조치 -> 감사 로그 확인`으로 고정합니다. (참조: `docs/architecture/admin-information-architecture.md`)
- 메뉴/탭 명명은 IA 고정값을 그대로 사용합니다: 8개 모듈(`Dashboard`, `Users`, `Community`, `Message`, `Operation`, `Billing`, `Analytics`, `System`)과 `Users` 상세 6탭(`프로필`, `활동`, `결제`, `커뮤니티`, `로그`, `관리자 메모`). (참조: `docs/architecture/admin-information-architecture.md`, `docs/specs/admin-user-detail-page-structure.md`)
- 조치 가능 화면에서는 "판단 정보"와 "실행 액션"을 같은 뷰에서 제공합니다. 단, 파괴적 조치 전에는 확인 단계와 영향 범위를 반드시 노출합니다.
- 감사 가능성(Auditability)을 UX의 1급 목표로 둡니다. 모든 변경 조치 후에는 해당 대상의 `감사 로그` 진입 경로를 즉시 제공합니다. (참조: `docs/specs/admin-action-log.md`)
- `시스템 로그`는 장애/기술 분석 용도이며 운영 조치 확인 용도의 `감사 로그`와 시각적으로도 분리합니다. (참조: `docs/architecture/admin-information-architecture.md`, `docs/specs/admin-action-log.md`)
- 상세 페이지에서는 `PageTitle` 하단 상세 정보 영역에 식별 가능한 핵심 값(예: 이름/이메일/ID)을 우선 노출합니다.
- 테이블 상세 확인 규칙: 연결 가능한 식별자(ID/작성자 등)는 링크 스타일 텍스트로 해당 페이지 이동을 우선 제공하고, 그 외 케이스는 행 클릭 시 `Descriptions` 기반 상세 모달을 표시합니다.

## 핵심 사용자 여정

운영자가 가장 자주 수행하는 핵심 여정은 아래 단일 체인으로 정의합니다.

```text
검색 -> 상세 -> 조치 -> 감사 로그 확인
```

- 검색: `Users > 회원 목록`에서 이메일/닉네임/상태/가입일/구독 상태 조건으로 대상을 좁힙니다. (필터 정의 참조: `docs/specs/admin-page-tables.md`)
- 상세: 목록 행 선택으로 `Users > 회원 상세`에 진입하고 상단 요약 카드에서 현재 상태를 1차 판단합니다. (구조 참조: `docs/specs/admin-user-detail-page-structure.md`)
- 조치: `프로필` 또는 연관 탭(`커뮤니티`, `결제`)에서 정지/해제/숨김/삭제 등 조치를 실행합니다.
- 감사 로그 확인: 조치 완료 토스트 또는 완료 패널의 딥링크로 `System > 감사 로그`를 열어 Action, Target ID, Time을 교차 검증합니다. (참조: `docs/specs/admin-action-log.md`)
- 성공 기준: 운영자는 한 세션 내에서 검색, 상세, 조치, 감사 로그 확인을 중단 없이 수행하고 증적을 남깁니다.

## 변형 여정

아래 여정은 핵심 체인을 유지하되 시작 페이지와 조치 유형만 달라지는 운영 시나리오입니다.

1) 신고 처리 여정 (`Community > 신고 관리`)
- 검색: 신고 상태 `처리 전`, 신고 사유, 기간으로 대상 신고를 찾습니다.
- 상세: 신고 항목에서 게시글 상세 컨텍스트를 확인합니다.
- 조치: `게시글 숨김` 또는 `사용자 정지` 중 하나를 실행하고 신고 상태를 `처리 완료`로 변경합니다.
- 감사 로그 확인: `System > 감사 로그`에서 `POST_HIDE`/`USER_SUSPEND` 계열 Action과 Target ID를 확인합니다.

2) 게시글 숨김/삭제 여정 (`Community > 게시글 관리`)
- 검색: 제목/작성자/게시판/신고 여부로 문제 게시글을 찾습니다.
- 상세: 게시글 행 클릭으로 상세보기 패널을 열어 본문과 신고 맥락을 확인합니다.
- 조치: 정책 위반 시 `게시글 숨김` 또는 `게시글 삭제`를 실행합니다.
- 감사 로그 확인: `감사 로그`에서 관리자 계정, Action, Time을 검증합니다.

3) 회원 정지/해제 여정 (`Users > 회원 목록` / `Users > 회원 상세`)
- 검색: `Users > 회원 목록`에서 이메일 또는 닉네임으로 대상 회원을 찾습니다.
- 상세: `Users > 회원 상세` 상단 요약 카드와 `로그` 탭으로 최근 로그인/행동 이력을 확인합니다.
- 조치: `회원 정지` 또는 `회원 정지 해제`를 실행합니다.
- 감사 로그 확인: `System > 감사 로그`에서 USER 단위 Target ID로 조치 이력을 확인합니다.

4) 메시지 발송 여정 (`Message > 메일` / `Message > 푸시`)
- 검색: 채널(메일/푸시), 자동/수동 탭, 대상 그룹으로 대상을 좁힙니다.
- 상세: 템플릿 미리보기(제목/본문/발송 그룹)로 오발송 위험을 점검합니다.
- 조치: `즉시 발송`, `예약 발송`, `나에게 보내기`, `자동 발송 활성화/비활성화`를 실행합니다.
- 감사 로그 확인: 발송 설정 변경/실행 이벤트를 `감사 로그`와 `Message > 발송 이력`의 상세 Drawer에서 교차 확인합니다. (참조: `docs/specs/admin-page-analysis.md`)

5) 결제 확인 여정 (`Users > 회원 상세 > 결제` / `Billing > 결제 내역`)
- 검색: `Users > 회원 목록` 또는 `Billing > 결제 내역`에서 회원/결제 ID를 찾습니다.
- 상세: `Users > 회원 상세 > 결제`에서 결제 상태와 결제 시점을 확인합니다.
- 조치: 기본 조치는 `결제 확인`이며, 환불/취소는 `Billing` 정책 플로우를 참조 수준으로 안내합니다.
- 감사 로그 확인: 결제 상태 변경이 발생한 경우 `감사 로그`에서 PAYMENT 계열 Action을 확인합니다.

## 화면 레이아웃 및 구성 (UI)

아래 레이아웃은 블록 단위(페이지 수준) 기준입니다. 컬럼/필터의 상세 표는 `docs/specs/admin-page-tables.md`를 참조하고 본 문서에는 중복 기재하지 않습니다.

### 1) Users > 회원 목록
- 상단: 페이지 타이틀, 총 건수, 빠른 상태 인디케이터
- 검색/필터 바: 검색어, 상태, 기간, 구독 조건, 초기화/저장 뷰
- 본문 테이블: 서버 사이드 정렬/페이지네이션/가상 스크롤 대응
- 우측 보조 패널(선택): 회원 미리보기와 빠른 조치 버튼
- 하단: 선택 건수, 대량 조치 진입점, 마지막 동기화 시간

### 2) Users > 회원 상세 (상단 요약 카드 + 탭)
- 상단 요약 카드: 이메일, 닉네임, 상태, 등급, 구독, 최근 로그인, 핵심 조치 버튼
- 탭 네비게이션(고정 6개): `프로필`, `활동`, `결제`, `커뮤니티`, `로그`, `관리자 메모`
- 탭 본문 영역: 탭별 데이터 블록 + 우측 조치 히스토리 블록
- 페이지 고정 액션 영역: 파괴적 조치 버튼과 `감사 로그` 바로가기

### 3) Community > 게시글 관리
- 상단: 게시글 운영 상태 요약(정상/숨김/신고 다발)
- 검색/필터 바: 제목/작성자/게시판/신고 여부/기간
- 본문: 게시글 리스트 + 인라인 상태 배지 + 빠른 조치(숨김/삭제)
- 상세 드로어: 게시글 본문, 신고 문맥, 작성자 정보, 조치 패널

### 4) Community > 신고 관리
- 상단: 미처리 신고 카운트, SLA 경과 경고
- 좌측 리스트: 신고 항목 큐(처리 전 우선)
- 우측 상세: 신고 사유, 대상 게시글, 신고자 정보, 유사 신고 이력
- 하단 액션 바: `게시글 숨김`, `사용자 정지`, `처리 완료`, `감사 로그` 확인 버튼

### 5) Message > 메일 / 푸시
- 상단: 채널 선택(메일/푸시)과 자동/수동 탭
- 본문: 템플릿 목록, 필터 바, 액션 메뉴(수정/미리보기/나에게 보내기/발송)
- 편집 모달: 기본 정보, 발송 그룹, HTML 본문, JSON 본문
- 완료 섹션: 발송 결과 요약, `발송 이력` 이동, 관련 감사 로그 링크

### 6) Message > 대상 그룹
- 상단: 검색과 정의 방식 필터, 신규 그룹 추가 버튼
- 본문: 그룹 리스트, 조건 요약, 예상 대상 수, 상태, 빠른 액션(수정/재계산/삭제)
- 우측 Drawer: 그룹명/설명/정의 방식/적용 채널/운영 상태와 세그먼트 편집, `조회하기` 기반 예상 발송 인원 계산
- 액션 영역: 저장/삭제 후 `메일`, `푸시`, `발송 이력`으로 이어지는 후속 이동 경로

### 7) Message > 발송 이력
- 상단: 채널 탭(메일/푸시), 보조 탭(전체/자동/수동), CSV 내보내기
- 본문: 발송건 리스트, 상태 필터, 검색, 발송 성공/실패 집계
- 우측 Drawer: 발송 기본 정보와 수신자 샘플 테이블, 수신자 검색, 재시도/복제 발송 버튼
- 액션 영역: `재시도`, `복제 발송`, `감사 로그` 확인 링크

### 8) System > 감사 로그
- 상단: 감사 로그 타이틀 + 필터 칩 + 기간 프리셋
- 본문 테이블: 관리자, Action, Target Type/ID, Time, Before/After 진입
- 상세 패널: 변경 전/후 비교, 관련 원본 페이지 딥링크
- 보조 안내: `시스템 로그`와 목적이 다름을 고정 문구로 표시

## 상호작용 모델

- URL 파라미터/필터 유지: 목록 화면의 검색/정렬/페이지/탭 상태는 URL 쿼리로 동기화하여 새로고침/뒤로가기/공유 시 동일 상태를 복원합니다.
- 목록-상세 왕복: 목록에서 상세 진입 후 돌아오면 기존 검색 결과와 스크롤 위치를 유지합니다.
- 대량 테이블 시나리오: 1,000건 이상 가능 화면(`Users`, `Community`, `감사 로그`)은 server-side 페이징/정렬/필터를 기본값으로 사용하고, 긴 목록은 가상 스크롤을 활성화합니다.
- 권한 기반 버튼 제어: 권한 없음 상태를 `숨김`, `비활성`, `실행 차단(서버 검증)` 3단계로 구분해 적용합니다.
- 조치 후 피드백: 모든 조치 완료 시 결과 토스트 + 대상 요약 + `감사 로그` 바로가기 액션을 제공합니다.

### Notification UX 규칙

- 노출 위치: 전역 우측 상단 `notification`을 사용합니다.
- 작동 시점: 맥락 정보(대상 유형/대상 ID/사유/감사 로그 링크)까지 함께 안내해야 하는 성공 이벤트에서 노출합니다.
- 콘텐츠 구성: 제목(조치명 + 완료), 본문(대상 유형, 대상 ID, 사유/근거), `감사 로그 확인` 이동 링크를 포함합니다.
- 중복 방지: 같은 조치 성공 이벤트에 대해 `message.success`를 동시에 노출하지 않습니다.
- 실패/주의 분리: 실패/검증 경고는 조치 실행 컨텍스트(폼, 모달, 인라인 경고)에서 우선 안내하고 성공 결과 알림과 구분합니다.
- 정합성: notification에 표시한 조치 정보는 감사 로그 테이블 필드와 동일 값으로 매핑되어야 합니다.

### Message UX 규칙 (기본)

- CRUD 성공 피드백의 기본 채널은 `message.success`입니다.
- 작동 시점: 조치가 실제 반영된 직후 1회 노출합니다.
- 콘텐츠: 한 줄 텍스트(예: `공지 수정 완료`)로 간결하게 유지합니다.

### Popconfirm UX 규칙 (Delete)

- Delete 확인의 기본 패턴은 `Popconfirm`입니다.
- 버튼 근처에서 즉시 확인/취소 가능한 짧은 경고 문구를 사용합니다.
- 사유 입력이 필요한 Delete는 `Modal`/`ConfirmAction`으로 전환합니다.

## 파괴적 액션 UX 규칙

- 대상 명시: 삭제/정지/숨김 모달에는 대상 식별자(닉네임/ID/게시글 ID)를 항상 표시합니다.
- 영향 범위 고지: 조치가 사용자 노출/권한/결제 상태에 미치는 영향을 한 줄 요약으로 노출합니다.
- 재확인 입력: 고위험 액션(탈퇴 처리, 게시글 삭제)은 확인 문구 입력 또는 2단계 확인을 요구합니다.
- 실행 책임 표시: 실행 주체(현재 관리자 계정)와 실행 시간이 감사 로그에 남는다는 점을 사전 고지합니다.
- 복구 경로 제공: 복구 가능한 조치(정지 해제, 숨김 해제)는 즉시 복구 버튼 또는 연계 경로를 제공합니다.

## 예외/오류/피드백

- 권한 오류(403): 버튼 클릭 시 차단 이유와 필요한 권한명을 안내하고 요청 경로를 제공합니다.
- 동시성 충돌(409): 다른 관리자의 선행 조치가 감지되면 최신 상태를 재로딩하고 비교 안내를 제공합니다.
- 대상 없음(404): 상세 진입 중 대상이 삭제/이동된 경우 안전 복귀 링크(목록, 감사 로그)를 제공합니다.
- 검증 실패(400): 필수값/형식 오류는 필드 인라인 메시지 + 요약 배너로 중복 안내합니다.
- 서버 장애(5xx): 조치 미완료를 명확히 표시하고 재시도/임시 저장/운영 채널 에스컬레이션을 안내합니다.
- 감사 로그 지연: 로그 반영 지연 시 "최대 N초 지연" 안내와 재조회 액션을 제공합니다.

- [ ] `감사 로그`와 `시스템 로그`를 명확히 구분했다.
- [ ] 핵심 여정 `검색 -> 상세 -> 조치 -> 감사 로그 확인`을 명시했다.
- [ ] 변형 여정에 신고 처리, 게시글 숨김/삭제, 회원 정지/해제, 메시지 발송/예약, 결제 확인(환불 참조)을 포함했다.
- [ ] `Users` 상세의 6탭(`프로필`, `활동`, `결제`, `커뮤니티`, `로그`, `관리자 메모`)을 고정값으로 명시했다.
- [ ] `Users > 회원 목록`, `Users > 회원 상세`, `Community > 게시글 관리`, `Community > 신고 관리`, `Message > 메일/푸시`, `System > 감사 로그`의 블록 레이아웃을 포함했다.
- [ ] URL 파라미터/필터 유지, 대량 테이블(server-side + 가상 스크롤), 권한 기반 버튼 제어 규칙을 포함했다.
- [ ] 컬럼/필터 표는 본문에 복붙하지 않고 `docs/specs/admin-page-tables.md` 등 근거 문서를 참조했다.
