# TOPIK AI Admin UX/UI 원칙

## 문서 계약

- 기준 모듈: Dashboard, Users, Community, Message, Operation, Commerce, Assessment, Content, Analytics, System
- 메뉴명은 `Users`를 코드 식별자로 유지하되, 사용자 노출 라벨은 역할에 맞는 한글 명칭을 사용합니다.
- 감사 추적 용어는 `감사 로그`로 통일합니다.
- Users 상세 탭은 `프로필`, `활동`, `결제`, `커뮤니티`, `로그`, `관리자 메모` 6개를 고정합니다.

## 라벨 언어 규칙

- 사용자에게 노출되는 메뉴, 브레드크럼, 페이지 제목, 탭, 버튼, 상태 라벨은 기본 한글로 작성합니다.
- 동일 개념을 영어와 한글로 혼용하지 않습니다. 예: `Message`와 `메시지`를 함께 남기지 않습니다.
- 영문 표기 예외는 제품명, 외부 서비스명, 코드/식별자, 법적 고유 명칭처럼 한글 치환 시 의미가 손상되는 경우로 제한합니다.
- 코드 식별자와 URL 경로는 영문을 유지할 수 있지만, 화면 라벨은 별도 한글 매핑을 둡니다.

## UX 배치

- 운영 기본 흐름은 `검색 -> 상세 -> 조치 -> 감사 로그 확인`입니다.
- 콘텐츠형 모듈은 `작성/수정 -> 확인 -> 배포 -> 감사 로그 확인` 흐름을 추가로 고려합니다.
- role에 따라 담당 범위가 달라도 같은 화면 안에서 모듈 경계와 책임이 명확해야 합니다.
- 아직 상세 기능이 없는 메뉴도 placeholder 화면에서 breadcrumb, 권한 안내, 예정 기능을 노출해야 합니다.

## 모듈별 초점

1. `Users`
- 회원 검색, 상세 확인, 상태 조치, 감사 로그 확인

2. `Message`
- 채널 선택, 템플릿 편집, 발송 실행, 발송 이력과 감사 로그 확인

3. `Commerce`
- 결제 조회, 환불 처리, 후속 정책 화면(쿠폰, 포인트, 이커머스)으로 이동

4. `Assessment` / `Content`
- 아직 상세 기능은 placeholder 상태지만 메뉴, 권한, 브레드크럼 경계를 먼저 노출

## 레이아웃 배치

- 목록형 페이지: `PageTitle -> FilterBar -> Table -> 후속 액션`
- 상세 drill-down: 링크 이동 또는 Drawer/Modal
- placeholder: `PageTitle -> 정보 Alert -> 권한 안내 -> 예정 기능 리스트`

## 상호작용 규칙

- 파괴적 액션은 확인 단계와 사유 입력 수단을 제공합니다.
- 성공 피드백은 대상 요약과 `감사 로그 확인` 경로를 함께 제공합니다.
- URL 파라미터로 목록, 필터, 탭 상태를 복원합니다.
